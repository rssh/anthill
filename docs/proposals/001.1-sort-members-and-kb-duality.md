# Proposal 001.1: Sort Members and the Record/KB Duality

**Status:** Partially Implemented (member facts, by_domain index, subst_term, load_all done; `import ... where` removed per 006, replaced by sort-level `requires`; inline binding evaluation deferred to resolve pass)
**Depends on:** [001-sort-domain-unification](001-sort-domain-unification.md)
**Affects:** Kernel Language Specification §2, §4.4, §5.2, §8.6, §11

## Motivation

Proposal 001 unified sorts and domains: a `sort X { ... }` declares both a type and a namespace. The grammar and parser implement this, but two key mechanisms remain unimplemented:

1. **Sort-level `requires`** — `requires Eq[T = T]` constrains sort instantiation (replaces the removed `import ... where` syntax; see proposal 006).
2. **Inline bindings** — `Option[T=Int64]` is parsed as a `ParameterizedType` but has no operational semantics.

Both require answering the question: *"does sort X have a member named T?"* The current KB representation makes this hard — a sort's members are scattered facts linked only by the `domain` field, invisible to the term-level unification engine.

This proposal addresses the representation gap by observing that **sort members are already KB facts**, and formalizes the duality between record-style and fact-style representations.

## The Duality

A sort with a body is equivalent to two representations:

**Intensional (record in term):**
```
Option = Record(Option, {T: Sort, none: Constructor, some: Constructor})
```

**Extensional (facts in KB):**
```
member(T, Sort, Option)
member(none, Constructor, Option)
member(some, Constructor, Option)
```

These carry the same information. The current KB already stores the extensional form — the `SortInfo` and `EntityOf` facts with `domain = Option` are exactly `member` triples. The insight is: **we do not need a new term kind for records**. The KB is the record store.

## The `member` Fact Convention

Today, sort structure is recorded as structured facts:

```
SortInfo(name: Option, definition: Option, constructors: [none, some],
         operations: [], parameters: [T], requires: [])   [sort: Sort, domain: _global]
SortAlias(T, ?)           [sort: Sort, domain: Option]    -- type parameter
EntityOf(none, Option)     [sort: Sort, domain: _global]
EntityOf(some, Option)     [sort: Sort, domain: _global]
```

We additionally produce **member facts** for each item declared directly in a sort body, providing a uniform index:

```
member(name, member_kind, parent_sort)   [sort: Member, domain: parent_sort]
```

Where `member_kind` is one of: `Sort`, `Constructor`, `Operation`, `Rule`, `Namespace`.

Example — `sort Option { sort T; entity none; entity some(value: T) }` produces:

```
member(T, Sort, Option)           [sort: Member, domain: Option]
member(none, Constructor, Option) [sort: Member, domain: Option]
member(some, Constructor, Option) [sort: Member, domain: Option]
```

The existing `SortInfo`, `EntityOf`, `Entity`, `Operation` facts remain (they carry additional detail). The `member` facts are an **index** — a uniform way to query "what does X contain?"

## Named Members vs. Logic Variables

Sort parameters are **named members**, not logic variables. This distinction is essential:

| | Named member | Logic variable |
|--|--|--|
| Syntax | `sort T` (inside a body) | `?T` (in terms) |
| Identity | By name: "T" | By binding position |
| Addressed via | `Option[T=Int64]` | Unification |
| Interchangeable? | No: `T` ≠ `U` | Yes: `?T` ≈ `?T1` |
| KB representation | `member(T, Sort, Option)` | None (ephemeral) |

When you write `Option[T=Int64]`, you address the member **by name**. This requires `T` to be a stable name in Option's interface, not a variable that could unify with anything.

Logic variables remain useful in rule bodies for pattern matching:

```
rule has_param(?S, ?Name) :- member(?Name, Sort, ?S)
```

Here `?S` and `?Name` are variables that range over KB facts. But `T` in `sort T` is a declaration — it creates a named member.

## Sort-Level `requires` Semantics

With the `member` convention, sort-level `requires` constraints become KB operations. The `import ... where` syntax has been removed (see proposal 006); instantiation is now expressed via `requires` at the sort level:

```
sort Monad {
  sort M
    sort T
  end
  requires Functor[F = M]
  operation return(x: T) -> M
  operation bind(m: M, f: T -> M[T = U]) -> M[T = U]
}
```

**Step 1 — Query:** Find all member facts in the required sort:
```
member(F, Sort, Functor)         -- Functor declares abstract sort F
member(map, Operation, Functor)
```

**Step 2 — Check compatibility:** For each member of Functor that is being bound (`F → M`), verify the target (`M`) satisfies the interface. Since `F` has a member `T`:
```
member(T, Sort, F)               -- F has parameter T
member(T, Sort, M)               -- M also has parameter T  ✓
```

**Step 3 — Substitute:** In the required sort's facts, apply the substitution `F → M` (using the existing `apply_subst` on terms). Assert the substituted facts into the current scope.

**Step 4 — Verify:** Run denial checking to ensure no contradictions arise.

This uses only existing machinery: KB queries, term substitution, and denial checking.

## Inline Binding Semantics

`Option[T=Int64]` is a **scoped specialization**:

1. Query: `member(T, Sort, Option)` — verify T exists as a member of Option.
2. Create a scoped context (a derived domain) where T is bound to Int64.
3. Within that scope, all references to T resolve to Int64.

In the KB, this can be represented as a derived fact:

```
bind(T, Int64)   [domain: Option_T_Int]   -- generated scope name
```

Or, equivalently, as a parameterized type term that the loader already parses:

```
ParameterizedType(Option, {T: Int64})
```

The key point is that `{T=Int64}` is a **named field access** operation, not unification. It looks up "T" in Option's member index.

## Interface Satisfaction as a Query

"Does X satisfy Y's interface?" is a KB query, not a term unification:

```
-- X satisfies Y if every member of Y exists in X
rule satisfies(?X, ?Y) :-
    member(?Name, ?Kind, ?Y),
    member(?Name, ?Kind2, ?X)
    -- optionally: Kind2 is a subkind of Kind
```

This is a Horn clause expressible in the kernel language itself. The matching logic is not baked into the unification engine — it lives in the KB as a rule, and can be extended, refined, or overridden.

For example, a stricter version could require kind compatibility:

```
rule strictly_satisfies(?X, ?Y) :-
    member(?Name, Sort, ?Y),
    member(?Name, Sort, ?X)

rule strictly_satisfies(?X, ?Y) :-
    member(?Name, Constructor, ?Y),
    member(?Name, Constructor, ?X)
```

## Impact on Existing Constructs

### Terms — unchanged

No new term kinds. `Fn`, `Var`, `Const`, `Ref`, `Bottom` remain as-is. Sort terms stay nominal (`Fn("Option", [])`).

### Unification — unchanged

First-order unification is not modified. Record matching is done via KB queries, not unification rules.

### Sort declarations — minor change

The loader additionally produces `member` facts for each direct body item. All existing facts (`SortInfo`, `EntityOf`, `Entity`, `Operation`) are still produced — `member` is an index layer on top.

### Sort-level `requires` — gain operational semantics

The `import ... where` syntax has been removed (proposal 006). Sort-level constraints are now expressed via `requires Spec{bindings}`. This proposal gives them meaning: query + compatibility check + substitution + re-assertion.

### `X[T=Int64]` inline bindings — gain operational semantics

Currently parsed as `ParameterizedType` but not evaluated. This proposal gives them meaning: named member lookup + scoped binding.

### Rules — can query sort structure

Rules can now match on sort interfaces via `member` facts:

```
rule parametric(?S) :- member(T, Sort, ?S)
rule has_constructors(?S) :- member(?C, Constructor, ?S)
```

## Example: Functor Specification

```
sort Functor {
  sort F
    sort T
  end
  operation map(c: F[T: ?A], f: ?A -> ?B) -> F[T: ?B]
}
```

KB after loading:

```
member(F, Sort, Functor)            -- Functor has member F
member(T, Sort, F)                  -- F has member T
member(map, Operation, Functor)     -- Functor has operation map
```

Instantiation via `requires` in a dependent sort:

```
sort Monad {
  sort M
    sort T
  end
  requires Functor[F = M]           -- M must satisfy Functor
  ...
}
```

1. Query members of `Functor` bound to `F` → `{F: Sort}`
2. Check: `member(T, Sort, M)` → ✓ (M has T)
3. Substitute F → M in all Functor's facts
4. Assert into current scope: `member(map, Operation, Monad)` with specialized signature

## Example: Monad Specification

```
sort Monad {
  sort M
    sort T
  end
  requires Functor[F = M]
  operation return(x: ?T) -> M[T: ?T]
  operation bind(m: M[T: ?A], f: ?A -> M[T: ?B]) -> M[T: ?B]

  rule bind(return(?x), ?f) = ?f(?x)
}
```

Instantiation — using `requires` to bind M to a concrete sort:

```
sort OptionMonad {
  requires Monad[M = Option]
}
```

Checks `member(T, Sort, Option)` → ✓. Substitutes M → Option. The operations `return` and `bind` are now specialized to Option.

## Relationship to ψ-terms

The extensional representation (`member` facts) is equivalent to Aït-Kaci's ψ-terms but without modifying the term algebra. In ψ-terms, a term carries labeled features and unification merges them. Here, the "features" live in the KB as `member` facts and "merging" is done by KB queries and fact assertion.

This avoids adding complexity to the unification engine while achieving the same expressive power. The trade-off: ψ-term unification is O(n) in feature count; KB queries depend on indexing. Since the KB already has domain-based indexing, this should be efficient in practice.

## Implementation Path

1. **Add `member` fact production to the loader** — extend `load_sort_with_body` and `load_namespace` to emit `member(name, kind, parent)` facts alongside existing facts.

2. **Implement sort-level `requires` processing** — query required sort's members, check compatibility, substitute, re-assert.

3. **Implement `X[T=Int64]` evaluation** — named member lookup and scoped binding.

4. **Add `satisfies` rule to the standard library** — expressible as a Horn clause over `member` facts.

Steps 1-2 are the core. Steps 3-4 build on them. No changes to the term algebra or unification engine are required.

## Backwards Compatibility

All existing `.anthill` files remain valid. The `member` facts are additional — they don't interfere with existing `SortInfo`, `EntityOf`, or other facts. Sort-level `requires` and `{T=Int64}` gain meaning where they previously had none.
