# Proposal 026.1: Value-integrated KB queries

**Status:** Draft
**Depends on:** [026-expression-evaluator](026-expression-evaluator.md)
**Related:** [010-query-system](010-query-system.md), [022-typing-as-facts](022-typing-as-facts.md)
**Affects:** `rustland/anthill-core/src/kb/{resolve,subst,discrim,mod}.rs`, `stdlib/anthill/reflect/`

## Motivation

Proposal 026 introduced a `Value` enum for the evaluator. KB resolution in `rustland/anthill-core/src/kb/` is still `TermId`-only:

- `kb.query(pattern: TermId)` (mod.rs:883)
- `kb.resolve(goals: &[TermId], ..)` (resolve.rs:798)
- `Substitution: VarId ‚Üí TermId` (subst.rs)
- `SearchStream` carries `Vec<TermId>` goals (resolve.rs:207)

Every reflect method with arguments is really a KB query with value inputs. Look at `stdlib/anthill/reflect/reflect.anthill`:

```anthill
operation sorts(kb: KB, namespace: Option[T = String]) -> List[T = SortInfo]
operation operations(kb: KB, sort_name: String) -> List[T = OperationInfo]
operation execute(kb: KB, query: LogicalQuery) -> Stream[T = Substitution, E = Error]
```

Each takes runtime `Value`s, must construct KB goals from them, and returns a `LogicalStream`. Today there is no contract between `Value` and the resolver ‚Äî either every such call hash-conses its inputs (and, symmetrically, rehydrates answers on the way out), or the resolver has to grow a path that speaks `Value` natively.

This proposal defines that contract.

## Requirements

1. **Reflect operations** with value arguments must resolve without ad-hoc "hash-cons everything" at each builtin.
2. **External-backed KBs** (DB-backed, streaming, file-sourced) must query without promoting row data into the hash-consed `TermStore` ‚Äî a 1M-row scan cannot grow the DB by 1M entries, since distinct-keyed rows never structurally dedupe.
3. **KB-resident data** must keep its current O(1) equality / cheap-clone properties ‚Äî ancestor-style recursive search over facts should not regress.
4. **Answer bindings** must be consumable as `Value` without a second projection pass.

## Design

### One input boundary, one output boundary

```
runtime Value  ‚îÄ‚îÄlower_query‚îÄ‚îÄ‚ñ∂  TermId goals  ‚îÄ‚îÄresolver‚îÄ‚îÄ‚ñ∂  VarId-to-Value bindings
                                                                    ‚îÇ
                                                                    ‚ñº
                                                           anthill code (Stream[Substitution])
```

**Input boundary** ‚Äî `execute(kb: KB, q: LogicalQuery) -> Stream[Substitution]` is the sole place runtime values are lowered to `TermId` goals. All reflect operations compile down to assembling a `LogicalQuery` and calling `execute`.

**Output boundary** ‚Äî none. Bindings are `Value`-typed from the moment they enter the substitution.

### `Substitution: VarId ‚Üí Value`

Storage changes from `HashMap<VarId, TermId>` to `HashMap<VarId, Value>`:

```rust
pub struct Substitution {
    bindings: HashMap<VarId, Value>,
    // ... parent chain, contradiction tracking unchanged
}

impl Substitution {
    pub fn bind_term(&mut self, v: VarId, t: TermId) {
        self.bindings.insert(v, Value::Term(t));   // fast path
    }
    pub fn bind_value(&mut self, v: VarId, val: Value) {
        self.bindings.insert(v, val);              // external-source path
    }
    pub fn lookup(&self, v: VarId) -> Option<&Value> { ... }
}
```

`Value::Term(tid)` is the dominant variant for resolver-internal bindings ‚Äî it carries a 4-byte `TermId` and preserves O(1) structural equality via hash-consing.

### `TermView` abstraction

A read-only interface implemented on `Value`:

```rust
pub trait TermView {
    fn head(&self, kb: &KnowledgeBase) -> Head;
    fn arg(&self, kb: &KnowledgeBase, i: usize) -> Option<ViewRef>;
    fn named(&self, kb: &KnowledgeBase, sym: Symbol) -> Option<ViewRef>;
}
```

`Value::Term(tid)` dispatches to `kb.get_term(tid)`; `Value::Entity/Tuple` read inline. Unification, pattern matching, and `match_term` become generic over `TermView` ‚Äî a single walker handles both KB-resident and external-sourced operands.

Candidate rule heads stay `TermId` (they live in the KB). The query side is where `TermView` bridges.

### Lineage-preserving bindings

**Rule:** Bindings inherit the representation of their source. Never materialize an existing `TermId` into an owning `Value::Entity`.

| Unifier reads from | Sub-field binding |
|---|---|
| `TermId` (KB-resident fact) | `Value::Term(sub_tid)` ‚Äî free, O(1) equality |
| `Value::Entity` (external stream row) | sub-`Value` passed through unchanged ‚Äî no promotion |
| `Term::Const(lit)` in rule body | `Value::Int/Bool/..` ‚Äî unbox once |

The anti-pattern this forbids is decomposing a `TermId` that's already hash-consed into a `Value::Entity` ‚Äî that does strictly more work (walks subterms, allocates SmallVec payloads) while making the binding heavier to clone at choice points.

### Query-lowering at `execute`

`execute_logical_query` walks the reified `LogicalQuery` and produces a goal list:

```rust
impl KnowledgeBase {
    pub fn execute_logical_query(&mut self, q: &Value) -> SearchStream {
        let goals = self.lower_query(q);
        self.start_search(goals, default_config())
    }

    fn lower_query(&mut self, q: &Value) -> Vec<TermId> {
        match q.as_entity_functor() {
            Some(sym::pattern_query) => vec![self.alloc_from_value(q.arg(0))],
            Some(sym::conjunction)   => { /* left ++ right */ }
            Some(sym::sort_query)    => { /* synthesize is_entity_of(?x, sort) */ }
            Some(sym::guarded)       => { /* q ++ [cond] */ }
            Some(sym::negation)      => vec![self.wrap_not(self.lower_query(q.arg(0)))],
            // ... other LogicalQuery constructors
        }
    }
}
```

`alloc_from_value` recursively promotes a `Value` into `TermId` by calling `TermStore::alloc`. This is the one place hash-consing is paid on the input side ‚Äî bounded by query *size*, not fact count.

## Scale: external-backed KBs

A DB-backed KB serves `LogicalStream<Value::Entity>` from cursor rows. With lineage-preserving bindings:

- Streaming iteration depth 1-2, branching ~1: substitution carries a few `Value::Entity` bindings; clone cost is dominated by the row itself, not the resolver.
- Unmatched rows are walked for structural check, then dropped ‚Äî `Value::Entity` storage is reclaimed with the row.
- Main `TermStore` grows only with facts/rules the user asserts, plus `execute`'s lowered query goals. **Never grows with streamed data.**

### Pathological case: deep recursion over external streams

A 10-deep recursive query against a 1M-row external source has heavy substitution cloning because each level carries `Value::Entity` bindings. Three escape valves ‚Äî none of them "hash-cons into main `TermStore`":

1. **Query-scoped arena.** Per-query promotion target separate from the main `TermStore`. Hash-consing happens, but only into disposable storage, dropped when the `SearchStream` is exhausted. Preserves O(1) equality during the query; no permanent DB growth.
2. **SharedStream memoization.** Cache prefixes of a stream consumed multiple times across backtracks. Covered by proposal 026 ¬ß"Stream variants (post-v1)".
3. **Query rewriting.** Push projections/filters into the stream source before recursion kicks in.

The `PromoteIfEntity` policy from earlier drafts is **not** adopted: it balloons the main `TermStore` with per-row entries that never structurally dedupe.

## Resolver changes

| File | Change |
|---|---|
| `subst.rs` | `Substitution::bindings: HashMap<VarId, Value>`. `bind_term` fast path, `bind_value` for external sources. `bind_compressed` operates on `Value::Term` entries (O(1) compare); falls back to structural when mixing variants. |
| `discrim.rs` | `SubstTree::query_resolved` generalizes to `impl TermView`. Tree internals unchanged ‚Äî only the walker differs. Candidate heads remain `TermId`. |
| `resolve.rs` | `SearchStream::stack` goals stay `Vec<TermId>` (rule bodies are KB-stored). Unification consumes a `TermView` for the substitution side; `bind_term` / `bind_value` chosen by source lineage. Query cache (`HashMap<TermId, ..>`) disables for non-`TermId` query terms. |
| `mod.rs` | Add `match_view<V: TermView>(pattern: TermId, target: V) -> Option<Substitution>` beside `match_term`. Add `execute_logical_query(q: &Value) -> SearchStream`. |

Builtins (`kb.execute_builtin`) keep their `TermId`-based signatures ‚Äî they inspect rule-head-side goals, which are always `TermId`. When a builtin reads a variable's binding it now reads a `Value`, which is usually simpler (direct scalar match instead of `Term::Const` dispatch).

## Reflect layer

The stdlib reflect module becomes thin:

```anthill
operation sorts(kb: KB, ns: Option[T = String]) -> List[T = SortInfo]
  -- desugars to:
  -- let q = conjunction(sort_query("_"), guarded(..., ns_filter(ns)))
  -- in execute(kb, q) |> collect
```

Each reflect method's body is "build a `LogicalQuery`, call `execute`, project the answer." No per-builtin value-to-goal conversion logic.

## Implementation milestones

**Q1 ‚Äî Substitution represents `Value`**

- `subst.rs`: rewrite `bind / lookup / compose / is_contradiction` against `Value`. Preserve `bind_compressed` fast path for `Value::Term` entries.
- Existing callers migrate via `bind_term`. No semantic change visible to tests.
- All `anthill-core` tests green.

**Q2 ‚Äî `TermView` abstraction**

- Trait + impls for `Value` and `(&KB, TermId)`.
- Generalize `SubstTree::query_resolved` and `match_term` ‚Üí `match_view`.
- Unification in `resolve.rs` consumes `TermView` on the binding side.

**Q3 ‚Äî `execute_logical_query`**

- `lower_query(&Value) -> Vec<TermId>` for every `LogicalQuery` constructor.
- Register as builtin backing `anthill.reflect.KB.execute`.
- Reflect operations (`sorts`, `operations`, `rules`, ‚Ä¶) rewritten to assemble `LogicalQuery` and dispatch through `execute`.

**Q4 ‚Äî External-source integration**

- Status: **landed** (WI-052 substrate + WI-052b resolver-side wiring). A goal whose head functor has a registered `RouteHandler` automatically pulls candidates from the handler's `ExternalStream` during `step_init` (see [007 ¬ß11 ‚Äî Q4 implementation contract (landed)](007-persistence-layer.md#11-value-integration-and-ingestion-contract-post-0261)).
- Originally filed as optional; the post-WI-168 unification with proposal 007's queryable-store path elevated it to **required for queryable persistence backends at scale**.
- `eval::stream::ExternalStream` Rust trait + `StreamSource::External(Box<dyn ExternalStream>)` variant. Yielded rows surface as `Value::Entity` and bind to œÉ via `bind_value` ‚Äî no `TermStore` allocation per row.
- Acceptance: a 10K-row external scan drains through `splitFirst` and leaves `KnowledgeBase::term_store_len()` unchanged (`anthill-core/tests/eval_q4_test.rs::q4_ten_thousand_row_scan_does_not_grow_term_store`).
- Example: in-memory `WorkItemRowStream` in the same test file demonstrates the canonical "row ‚Üí `Value::Entity`" shape. Future backends (file parser, GitHub issues, REST API) plug into the same trait ‚Äî only the row source differs.
- Threshold guidance: bulk pull is acceptable below ~10K rows in a queried table; Q4 is required at or above that scale, or for unbounded streams.

**Q5 ‚Äî Query-scoped arena (optional, profile-driven)**

- Arena whose `TermId`s are tagged separate from main `TermStore`.
- Policy hook on `bind_value` for deep-recursion-over-streams pattern.
- Gated on a concrete workload that profiles show needs it.

## Non-goals

- Changing the public anthill query surface (the `LogicalQuery` ADT in `reflect.anthill` is unchanged; this proposal is about its execution).
- Multi-store federation across arenas (each query runs against one main KB + at most one scoped arena).
- Incremental / cached query results across KB mutations.

## Open questions

1. **Substitution equality under lineage.** Two bindings `Value::Term(tid)` and `Value::Entity{..}` for structurally equal terms compare equal today (under proposal 026 ¬ßEquality). Does the resolver need this to be O(1), or is the occasional structural compare acceptable? Likely fine ‚Äî cross-variant compares are rare.
2. **`bind_compressed` under `Value`.** Path compression chases `Value::Term(tid)` bindings recursively. If a chain passes through a `Value::Entity`, compression halts at that node. Acceptable, but needs documenting as an invariant.
3. **Reflect method dispatch.** Each reflect op can either be (a) a Rust builtin that constructs `LogicalQuery` + calls `execute`, or (b) an anthill-level rule that does the same. Prefer (b) where the body is pure assembly ‚Äî fewer Rust touchpoints when the ADT evolves.
