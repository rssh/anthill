## Motivation

Three neighbouring proposals leave exactly one thing unbuilt between them:

- **027** gives `Branch` as `branch(a, b)` / `fail()` — a **two-way** choice interpreted as resolver
  choice points (LIFO trial order via `push_choice`).
- **047** frames the denotation (`Branch ↦ Stream`) and the mechanism (`reflect`/`reify`), and its §9
  build path step 3 says Branch "rides the existing lazy-choice substrate — mostly wiring."
- **052** makes a relation a first-class **stream** of typed solutions (`LogicalStream[{board}]`),
  consumed purely (`map`/`fold`/`splitFirst`).

What is missing is the operation that connects them: **`reflect(stream)` — lift an *arbitrary* solution
stream into the `Branch` effect** (n-way choice, generalizing 027's two-way `branch`), so a solution's
body runs in **direct style** and may itself **re-enter search**; plus the **`reify` side as
first-class solver *values*** (`all`/`one`/`beam` as `Branch` handlers, not keywords); plus the one
genuinely-new runtime piece — the **eval↔SLD switch** that suspends the concrete evaluator to drive
SLD and multi-shot-resumes the continuation per solution.

052 is the pure-stream layer (a solution set is a value). This proposal is the effect layer: the same
`match`/`let` surface, but where a branch's body is ordinary direct-style code that can call back into
search. The boundary between the two proposals is **nesting** — flat consumption is 052; a body that
re-enters search needs the `Branch` effect and the switch, here.

## Design

### `reflect(stream)` — a stream becomes branches

The bridge operation lifts any `LogicalStream[T]` into ambient nondeterminism:

```anthill
operation reflect(s: LogicalStream[T]) -> T effects Branch
```

Each element of `s` becomes a branch; the empty stream is `fail()`. This is the `reflect` side of
047's reflection (`reflect : M a -> a` at `M = Stream`), and it **generalizes 027's `branch`**:

```
branch(a, b)  ≡  reflect(stream_of(a, b))     -- two-way is the 2-element stream
fail()        ≡  reflect(empty)               -- fail is the empty stream
```

Because it lifts *any* stream, rules are not privileged — a 052 relation-stream, a data stream, and a
literal `stream_of(...)` all reflect identically. Under the hood, `reflect` registers the stream's
elements as resolver choice points (`push_choice`, 027 §RuntimeAPI / 033); it adds no new choice
substrate, only the n-ary entry point over `SearchStream`.

### Solvers are `reify` handlers, as values

047 §7 makes a handler an ordinary value installed with `provide`; here each **solver** is a `Branch`
handler — the `reify`/`reset` boundary — reified as a first-class value:

```anthill
sort Solver
  operation run[T](goal: Goal) -> LogicalStream[T]    -- "provides matching": reify the Branch region

fact Solver[= all]            -- collect every solution         (047 observeAll; 027 default handler)
fact Solver[= one]            -- the unique solution — ι iota    (undefined if not exactly one)
fact Solver[= once]           -- first-commit (take any one)
fact Solver[= beam(k, score)] -- best-first / scored            ← 027.3 ScoredBranch, as one instance
fact Solver[= oracle(xs)]     -- pre-supplied choices
```

A `Solver` instance = a `Branch` handler = a LogicT observer = a stream consumer — one thing under four
names. **`beam` is not new machinery**: it is exactly [027.3-scored-branch-effect](027.3-scored-branch-effect.md)'s
`ScoredBranch` (priority-queue-backed `scoredMplus`/`multiScore`, best-first resume) surfaced as a
solver value. Users can define their own solver with no grammar change. The difference from 052's
consumers of the same names is *only* the reading: there `all`/`one` are pure stream folds; here they
are `Branch` handlers, so the region they reify may `reflect` again inside a body (nested search).

### Surface — `match <solver> case <goal>` / `let <goal> = <solver>`

Reusing `match`/`case` and `let` verbatim (no new grammar); the solver is a scrutinee value, the goal
is solved:

```anthill
match all  case queens(board)     -> body(board)   -- body per solution; may re-enter search
match one  case queens(board)     -> body(board)   -- the unique (iota)
match beam(k, cost) case path(?p) -> ...           -- best-first (027.3)

let queens(board) = all           -- board bound per solution (Branch); chains as conjunction
let member(x, xs) = all
... body(x, board) ...
```

Both forms desugar to:

1. evaluating `<goal>` performs `reflect` on the goal's solution stream (the `Branch` effect);
2. each solution binds **lexically** through the `case`/`let` pattern (logic vars stay at the
   metalevel, 052 §"logic variables");
3. the **`Solver`** is the enclosing `reify`/`provide` boundary that consumes the branches (collect /
   take-unique / priority-pull). A `Branch` with no enclosing solver is **unhandled ⇒ a loud
   load/type error** (047 §7); nondeterminism never leaks past a consumer.

Chained `let`s are conjunction (each binds vars the next uses); multiple `case`s are disjunction.

### The eval↔SLD switch — the one new runtime piece

Step 1's producer is the genuinely-new runtime mechanism: **suspend the concrete evaluator at the
goal, drive SLD, and multi-shot-resume the continuation once per solution.** The substrate is nearly
free because the interpreter is *already* suspend/resume-ready (047 §4): its `ActivationStack` is
non-native (`run()` trampolines over an explicit heap stack, so the continuation is reified data).
What is missing is small and mechanical:

- a **`StepOutcome::Suspend`** that returns from `run()` with the stack intact plus a resume entry
  (the same shape 047/WI-455 plans for `StepOutcome::Dispatch`);
- **`Clone` on `Frame`/`ActivationStack`** for multi-shot resume — **already committed (WI-078
  Phase-A/a)**;
- **coordinated binding**: the interpreter keeps its suspended stack; the resolver frame holds only a
  *resume handle* it calls back — so the resolver never embeds eval types (the eval↔resolve decoupling
  stays intact). A solution's binding re-enters the resumed eval by variable identity (WI-246 makes
  the value flow direct).

Concrete values already flow *into* SLD (WI-246 `Value`-carrying goals); choice points already exist
(`push_choice`/`SearchStream`, 033 / WI-075 Verified). The switch is what connects the evaluator to
them — 047 §5's "Suspension snapshot/resume is the one genuinely-missing piece," scoped to `Branch`.

### State across branches — 047 §8 effect ordering

Mutable state (KB facts, `Cell`s) under backtracking is governed by 047 §8: a resource ranked **below**
`Branch` survives backtracking (an audit/trace log); ranked **above**, it is undone on backtrack (a
speculative log). `register_undo` (027 §RuntimeAPI) is the ranking mechanism, not bespoke machinery.
Read-only search sidesteps it entirely; it bites only for *mutating* effects performed inside a branch.

**`External` state ([054](054-external-effect.md)) admits *neither* rank** — there is no
`register_undo` for the world, and below-`Branch` means one execution per multi-shot resume — so a
`Branch` region may not perform `External` (054 §"`Branch` and `External`"): hoist external reads
before the search, defer external writes to after the solver commits. A **fake** of an external
entity (054 §"Faking") is exactly the `Modify`-class substitute that restores `Branch`-compatibility:
its registry is tracked state under the ordinary snapshot contract, which is what lets a test *search*
over protocol schedules against it — `oracle(xs)` replays one interleaving, `all` enumerates a finite
schedule space and checks an invariant across every branch.

### NotFound in the effectful reading

The empty solution set is the `nil` arm of the stream match (052 §NotFound), surfaced by the solver
that reifies the region: `all` yields an empty stream (the `nil`/"found nothing" fallback); `one`
(iota) on empty yields the `nil` arm, or `Option`/`Error[NotFound]` per the site's policy. Because the
solver is the `reify` boundary, it is the natural place to decide the empty-region result.

## Relationship to neighbouring proposals

- **027** is the base: `Branch`, the `HandlerAction` carrier (`Choice`/`Fail`), `push_choice`
  interpretation. This proposal adds the n-ary `reflect(stream)` entry point over the same carrier and
  makes the reify side a first-class solver value.
- **047** is the semantic frame: `reflect`/`reify`, `Branch ↦ Stream`, effect order §8, and the "non-
  native stack is the design purpose" that makes the switch cheap. This proposal is the concrete
  Branch-from-streams instance of 047 §9 step 3.
- **052** supplies the typed streams to reflect (a relation as `LogicalStream[NamedTuple]`). Same
  surface, split by nesting.
- **027.3** is the `beam`/scored solver instance — reused, not re-specified.

## Build path

Mirrors 047 §9 step 3, concretized for `Branch`-from-streams:

1. **`reflect(stream): T effects Branch`** over the existing lazy-choice substrate — register a
   stream's elements as choice points (`push_choice`); `branch`/`fail` become the 2-element / empty
   special cases. *(Mostly wiring on 033 / WI-075.)*
2. **The eval↔SLD switch** — `StepOutcome::Suspend` + multi-shot resume + coordinated binding
   (`Clone` from WI-078 Phase-A/a; snapshot/resume WI-069). The one real interpreter addition.
3. **Solvers as reify handlers** — `all`/`one`/`once`/`oracle` as `Branch` handlers installed via
   `provide`; `beam` = 027.3 `ScoredBranch`.
4. **Surface desugar** — `match <solver> case <goal>` / `let <goal> = <solver>` → `reflect` + solver
   handler, reusing `case`/`let` binders.

Step 1 is largely wiring; step 2 is the sole genuine interpreter change; steps 3–4 add capabilities on
the same substrate.

## Open questions

1. **Coordinated-binding seam.** The exact handshake by which a resumed eval receives a solution's
   binding — WI-246 variable identity should make it direct, but the resume-handle ownership between
   interpreter and resolver frame needs pinning (who holds the suspended stack, who re-enters).
2. **`reflect` cost vs. 027's specialized `Choice` carrier.** Confirm `reflect(stream)` takes the cheap
   `push_choice` path (047's specialized carrier) rather than a general `shift`/`reset` for the common
   finite-stream case.
3. **Solver `run` signature / the observer split.** `run(goal) -> LogicalStream[T]` vs. an
   `Observer`-style incremental interface (047 / `CpsLogicMonad`'s `observeN`); how the `case`
   pattern's projection `T` threads through.
4. **Nested `provide` scope vs. 047 §8 rank.** For two nested searches with interacting state, confirm
   the rank table (not `provide` nesting) fixes the order, per 047 §8.
5. **`one` degeneracy (>1).** Whether "more than one solution" for `one`/iota is a runtime error, a
   `case multiple` arm, or silently first-commit (then it is `once`). Default: `one` = strict iota.
