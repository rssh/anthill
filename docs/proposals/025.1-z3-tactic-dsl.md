# Proposal 025.1: Z3 Tactic DSL

**Status:** Draft
**Depends on:** [025-proof-constructs](025-proof-constructs.md)
**Related:** [026-expression-evaluator](026-expression-evaluator.md), [023-kb-guards](023-kb-guards.md)
**Affects:** `rustland/anthill-smt-gen/`, proof-grammar `field_list` semantics inside `by z3(...)`

## Motivation

Proposal 025 introduced `proof name by z3(logic: "LRA") end` as the surface form for SMT-discharged obligations. The argument list inside `by z3(...)` was intentionally underspecified — only `timeout` and `logic` were called out. Real proofs need more.

The lf1 transponder follower (`examples/webots-modelling/lf1/safety_transponder.anthill`) is the immediate driver. Its `d_max` upper bound is **not a per-step inductive invariant** — extremum-seeking yaw flips can transiently drive the follower outward while it searches for the gradient. The honest proof is *bounded-excursion*: from any post-armed state, the controller forces a yaw flip within at most `N` consecutive bad ticks. That requires a **ranking function** (well-founded measure) plus a strict-decrease lemma — two SMT queries assembled by an outer tactic, not a single satisfiability check.

Today this proof exists as a hand-written test (`anthill-smt-gen/tests/lf1_real_spec_test.rs::lf1_transponder_excursion_ranking_function_manual`). The fact `PostArmedExcursionBound(max_ticks: 6)` is plugged into `safety_transponder.anthill` as a non-derived fact — the meta-theorem the test proves is bolted on by hand. This proposal lifts the manual discharge into a first-class Z3 tactic so the meta-theorem becomes an in-language proof.

The same mechanism handles other proof shapes that don't fit a single `(check-sat)`:

- **Induction** over an inductive sort (auto-emit case-split + IH-using subgoals).
- **Bounded model checking** for safety properties of step-bounded systems.
- **Tactic combinators** (`then`, `or_else`, `repeat`) so users can express fallback strategies without leaving the proof grammar.
- **Per-tactic outcome handling** — counterexamples become facts, unsat cores attach as proof metadata, partial results are inspectable.

## Design

### Three layers inside `by z3(...)`

The argument list to `by z3(...)` decomposes into three concerns. Most proofs touch only the first one.

| Layer | Concern | Example arguments |
|-------|---------|-------------------|
| **Translation** | anthill KB → SMT-LIB | `logic`, `mapping`, `quantifier` |
| **Tactic** | Z3-side proof strategy | `tactic`, `induction`, `ranking`, `bmc`, `assume` |
| **Outcome** | what to do with the result | `timeout`, `model`, `cores`, `interpolation` |

Layer separation is not enforced syntactically — all three appear as flat keyword arguments inside the same parenthesised list. The grouping is documentation; the implementation reads them in any order.

The current syntax `by z3(logic: "LRA")` is a tactic-layer shorthand for `by z3(tactic: smt(logic: "LRA"))`. This proposal preserves it — adding a `tactic:` argument is purely additive.

### Tactic catalog

Each tactic value is a term — either a bare identifier (`smt`, `induction`) or a function-application form with named arguments (`smt(logic: "LIA")`, `induction(over: TransponderFollowerState)`). Tactic terms compose via combinators (`then`, `or_else`, `repeat`).

#### `smt` — the default

Single satisfiability check. Equivalent to today's behaviour.

```anthill
proof lower_violation
  by z3(tactic: smt(logic: "LRA"))      -- explicit form
end

proof lower_violation
  by z3(logic: "LRA")                   -- shorthand: tactic is `smt` by default
end
```

#### `ranking` — bounded-excursion via well-founded measure

Discharges a *bounded-excursion* obligation by emitting two satisfiability queries:

1. **Boundedness**: the measure is non-negative on every reachable state.
2. **Strict decrease**: every `bad_step` transition strictly decreases the measure.

```anthill
proof bounded_excursion_transponder
  by z3(tactic: ranking(
    measure:        neg_upc,                     -- a rule: ?r = neg(?state.upc)
    decrease_when:  bad_step,                    -- a rule: bad_step(?s, ?s')
    state_sort:     TransponderFollowerState,
    well_founded_on: Int,                        -- the codomain of measure
    logic:          "LIA"
  ))
end
```

Required arguments:
- `measure` — name of a single-valued rule `measure(?state, ?r)` (or `?r = ...(?state)`). The codomain must be well-founded under the standard order.
- `decrease_when` — name of a binary predicate `bad_step(?state, ?next_state)` over the same `state_sort`.
- `state_sort` — the sort the measure and predicate range over. Used to fix Z3 declarations.
- `well_founded_on` — the measure's codomain (today: `Int` or `Nat`; future: any sort with a declared `WellFounded` provides-fact).

The tactic emits both queries with a shared SMT-LIB preamble (sort declarations, fact assertions). The outer conclusion — *"every `bad_step` chain has length ≤ measure(initial)"* — is recorded as a derived fact:

```anthill
fact RankingProof(
  name:           bounded_excursion_transponder,
  measure:        neg_upc,
  bad_step:       bad_step,
  bound_witness:  6                        -- max measure value over the precondition
)
```

This fact replaces the hand-plugged `PostArmedExcursionBound(max_ticks: 6)` in `safety_transponder.anthill`. Downstream rules (`excursion_distance_bound`) read the bound from the proved fact instead of an axiomatised number.

#### `induction` — structural induction over an inductive sort

Auto-emits case-split + IH subgoals using the auto-generated induction principle (proposal 025 §"Auto-generated induction rules"). Each subgoal is itself an SMT query.

```anthill
proof list_length_nonneg
  by z3(tactic: induction(
    over:    List[T = ?T],
    base:    smt(logic: "LIA"),                  -- discharge the base case
    step:    smt(logic: "LIA")                   -- discharge the inductive case
  ))
end
```

Defaults: `base` and `step` both default to `smt` with the proof-statement's inferred logic. Specifying them is only required when the cases need different tactics (e.g. base is decidable in LIA but step needs LRA).

Verdict mapping for `bmc` is *different* from the other tactics:

| Z3 result | Outcome status | Why |
|-----------|----------------|-----|
| `unsat`   | `bmc_clean(depth: k)` | no counterexample up to k steps; **not** `proved` |
| `sat`     | `disproved` (with counterexample trace) | real counterexample at depth ≤ k; sound |
| `unknown` | `unknown` | timeout / solver gave up |

The `bmc_clean` status is a separate point in the lattice — it is *not* a flavour of `proved`. Rules consuming a `bmc_clean` outcome must explicitly mention the depth:

```anthill
rule transponder_safe_for_short_runs(?depth)
  :- BmcCleanResult(proof: transponder_no_collision_within_100_ticks,
                    depth: ?depth)
```

A `proof` that claims unbounded safety **cannot** be satisfied by a `bmc` tactic. Attempting it (i.e. setting `by z3(tactic: bmc(...))` on a proof whose statement quantifies over all reachable states) is a load error: the tactic's strength is statically weaker than what the obligation demands. The error message points the user at either (a) reformulating the statement to quantify over depth, or (b) adding an inductive `tactic: induction(...)` or `tactic: ranking(...)` for the unbounded case.

Combinator interaction: `or_else(bmc(...), induction(...))` is well-typed *only if* the BMC branch's success would also satisfy the obligation — which by the rule above means the obligation must be the bounded form. For unbounded obligations, `or_else(induction(...), bmc(...))` is the only ordering that makes sense (induction is the actual proof; BMC is a fallback that produces evidence rather than a proof, and the user has to acknowledge that explicitly).

The combinator runtime returns the strongest verdict obtained. `or_else(induction, bmc)` returning `bmc_clean(100)` after induction failed means: "we couldn't prove this; we've checked 100 steps and found nothing." That's useful information, distinct from `proved`.

#### Combinators: `then`, `or_else`, `repeat`

Composition for fallback strategies. Mirrors Z3's native tactic combinators, surfaced verbatim because the semantics are well-known.

```anthill
proof step_distance_bound_under_two_meters
  by z3(tactic:
    or_else(
      smt(logic: "LRA"),                         -- try cheap LRA first
      smt(logic: "NRA")                          -- fall back to nonlinear
    ))
end

proof some_complex_property
  by z3(tactic:
    then(
      simplify,                                  -- preprocess
      induction(over: List, step: smt(logic: "LIA"))
    ))
end

proof eventually_holds
  by z3(tactic: repeat(unfold(of: my_recursive_rule), times: 5))
end
```

`simplify` here is **Z3's native simplify** — it rewrites Z3's own AST using Z3's built-in rules (`(* 0 x) → 0`, `(or true x) → true`, etc.). It does *not* see anthill's equational rules (`rule add_zero(?x) = ?x`, etc.); those live in the KB and aren't translated to Z3 axioms by this tactic. See "Anthill-rule-aware simplification (deferred)" below for the separate concern.

`unfold` is similarly Z3-side — it expands a Z3 `(define-fun ...)` definition. To unfold an anthill rule, that rule first needs a `(define-fun ...)` translation, currently obtainable only via the `mapping ... -> rule X` mechanism (per-symbol).

Both `simplify` and `unfold` are translation-layer tactics — they rewrite the SMT-LIB before handing it off. Listed here because they compose like proof tactics; arguably they belong to layer 1.

### Relationship to SMT-LIB and Z3 native facilities

A reasonable critique of this proposal is: *"SMT-LIB and Z3 already have a tactic language; why are we inventing a parallel one?"* The honest answer is that the overlap is real but partial — and acknowledging it changes some of the design.

#### What Z3 already has that we should pass through, not reinvent

Z3's input language extends SMT-LIB with a rich tactic algebra. Combinators (`then`, `or-else`, `par-or`, `repeat`, `try-for`, `if`, `when`, `using-params`) and tactics (`simplify`, `smt`, `qe`, `nlsat`, `qfnra-nlsat`, `propagate-values`, `propagate-ineqs`, `cofactor-term-ite`, `aig`, `bit-blast`, `tseitin-cnf`, …) are first-class. `(check-sat-using <tactic>)` invokes them.

Tactics and arguments that map directly onto Z3 native:

| 025.1 surface | Z3 native | Disposition |
|---------------|-----------|-------------|
| `tactic: smt(...)` | `(check-sat-using smt)` | pass-through |
| `tactic: simplify` | tactic `simplify` | pass-through |
| `tactic: then(a, b)` | `(then a b)` | pass-through |
| `tactic: or_else(a, b)` | `(or-else a b)` | pass-through but with anthill-side outcome combination (see below) |
| `tactic: repeat(t, times: 5)` | `(repeat t 5)` | pass-through |
| `tactic: par(a, b)` | `(par-or a b)` | pass-through |
| `logic: "LRA"` | `(set-logic LRA)` | pass-through |
| `timeout: 5000` | `(set-option :timeout 5000)` | pass-through |
| `model: true` | `(set-option :produce-models true)` + `(get-model)` | pass-through |
| `cores: true` | `(set-option :produce-unsat-cores true)` + `(get-unsat-core)` | pass-through |
| `interpolation: true` | `(set-option :produce-interpolants true)` + `(get-interpolant ...)` | pass-through |
| `mapping ... -> ...` | `(define-fun ...)` for renamings | pass-through |

For these, the implementation should serialise directly to Z3 input rather than writing our own algebra. A pass-through-friendly backdoor lets users drop into raw Z3 tactics when they want:

```anthill
proof some_obligation
  by z3(tactic: raw("(then simplify (using-params smt :random_seed 42))"))
end
```

`raw` is a tactic-position escape hatch — its body is a Z3 tactic expression spliced verbatim into `(check-sat-using ...)`. Useful when the user knows a Z3 incantation that solves their problem and we don't want to grow the catalog to cover it.

#### What Z3 doesn't have that we genuinely add

Three of the catalog entries don't map onto Z3 tactics — these are anthill-side **meta-tactics** that *expand to multiple* SMT-LIB queries each potentially using Z3 native tactics inside.

- **`ranking`.** Z3 has no `ranking` tactic. The well-founded-measure pattern is done by hand: the user emits two satisfiability queries (boundedness, decrease) and combines verdicts. This is exactly what the lf1 `lf1_real_spec_test.rs::lf1_transponder_excursion_ranking_function_manual` test does today. The 025.1 `ranking` tactic packages that — it's an anthill-layer construct that *uses* Z3 (via `smt`) for its sub-queries.

- **`induction`.** Z3 has *experimental* structural-induction support (`(set-option :induction true)`) but it's narrow and brittle. For induction over anthill's `enum`/`sort` definitions, we own the sort structure (constructors, field types) — we should generate the case-split + IH subgoals at the anthill layer where we have full type information, then dispatch each subgoal to Z3 via `smt`. The Z3 native form is not a viable substitute.

Both tactics also touch the **outcome layer** in a way Z3 cannot: they emit `RankingProof` / `ProofCore` facts back into the KB. A `(check-sat-using ...)` call only returns sat/unsat; it can't write our facts. So even where the inner sub-queries are Z3-native, the outer assembly stays anthill-side.

#### The CHC angle: Spacer is the right backend for transition-system safety

Anthill rules are Horn clauses. Z3 ships a CHC (Constrained Horn Clauses) engine called Spacer (`(set-logic HORN)` with the `(query ...)` command) that is *purpose-built* for safety-property verification of Horn systems. Spacer's native capabilities cover most of what 025.1's anthill-side meta-tactics try to do:

| Anthill meta-tactic | Spacer equivalent |
|---------------------|-------------------|
| Inductive-invariant synthesis | IC3/PDR — built-in |
| `ranking(measure, decrease_when, ...)` | μZ "termination" / "liveness" mode (extension) |
| Bounded model checking (deferred from 025.1) | Spacer with depth bound — built-in |

This suggests a fourth tactic — `chc` — that translates the anthill safety obligation into HORN-logic and dispatches to Spacer:

```anthill
proof transponder_safe
  by z3(tactic: chc(
    transition: step,
    initial:    initial_state,
    bad:        d_min_violated
  ))
end
```

`chc` is *sound* (Spacer's IC3 is a real proof procedure, not a bug-finder when it returns "safe"). When successful it produces `proved` and emits the synthesised inductive invariant as a derived rule — usable downstream for free.

The relationship to the other tactics:

- `chc` likely subsumes the safety-flavoured uses of `induction` + simple `ranking` cases. We keep `induction` / `ranking` because (a) `chc` is heavyweight and `smt` is enough for many obligations; (b) `induction` over anthill's `enum`-defined inductive sorts is structural, which is a different shape from CHC's transition-system framing; (c) `ranking` lets the user *supply* the measure when Spacer's automated synthesis fails.
- For lf1 specifically: the bounded-excursion proof could be re-tried as `chc(transition: step, initial: post_armed_state, bad: ...)` with no need for a hand-supplied measure. If Spacer's termination analysis succeeds we'd discover N=6 (or a tighter bound) automatically, and the `PostArmedExcursionBound` fact becomes Spacer's certificate rather than our hand-plug.

A follow-up proposal can flesh out `chc` once a translation pass to HORN logic is implemented; for now 025.1 calls it out as the architectural sibling that should constrain the `ranking` / `bmc` / `induction` design (we shouldn't reinvent what Spacer does well).

#### What this means for the catalog

Revised positioning of each tactic, after considering existing facilities:

| Tactic | Lives at | Rationale |
|--------|----------|-----------|
| `smt` | thin wrapper over Z3 `check-sat-using smt` | pass-through |
| `simplify`, `qe`, `propagate-values`, … | thin wrappers over Z3 native tactics | pass-through |
| `then`, `or_else`, `repeat`, `par` | anthill-layer wrappers around Z3 combinators | combinators serialise directly to Z3, but the *outcome* combination (which fact gets emitted, which staleness is recorded) is anthill-side |
| `raw("...")` | escape hatch | lets users drop to native Z3 tactic syntax when the catalog is too narrow |
| `induction` | anthill-side multi-query | uses Z3 only for sub-queries; the case-split structure comes from anthill's inductive-sort definitions |
| `ranking` | anthill-side multi-query | the dual queries are emitted by us; Z3 only sees `smt` calls |
| `chc` (future) | thin wrapper over Spacer | sound safety proofs for transition systems; replaces hand-rolled BMC unrollings |

The honest framing: 025.1's tactic DSL is *thin* over Z3 wherever Z3 already has the construct, *anthill-layer* only where the work involves multi-query assembly or KB-fact emission. The "DSL" is more accurately a small façade over Z3's tactic language plus two meta-tactics (`induction`, `ranking`).

### Anthill-rule-aware simplification (deferred)

A reasonable expectation: when I write `tactic: simplify`, my anthill rewrite rules (`rule add_zero(?x) = ?x`, `rule mul_one(?x) = ?x`, etc.) should kick in — Z3 should know about them and use them.

025.1 deliberately does *not* do this. Reasons:

- **Not all equational rules are simplifications.** Anthill admits `rule lhs = rhs` for many shapes:
  - Directional rewrites that always shrink terms: `rule add_zero(?x) = ?x` ✓
  - Equalities that would loop as rewrites: `rule add_comm(?x, ?y) = add(?y, ?x)` ✗
  - Definitions that go either way: `rule isEmpty(?s) = eq(length(?s), 0)` — direction depends on context
  - Operational specs that are properties, not rewrites: `rule push_pop(?s, ?x) = ?s`

  Translating *every* equational rule into a Z3 axiom (or anthill-side rewrite) without knowing the user's intent will at best blow up search-space, at worst introduce non-termination (`add_comm` applied as a rewrite loops forever).

- **A "simp set" concept doesn't exist in anthill yet.** Mathlib / Lean uses `@[simp]`, Coq uses `Hint Rewrite`, Isabelle uses `[simp]`. All have an explicit author-side opt-in marking which equalities are intended as rewrites. Anthill needs the equivalent before bulk translation makes sense, and the marking is its own design question (annotation? dedicated `simp_rule` keyword? `provides Simp` declaration? namespace-scoped sets?). That belongs in a separate proposal.

What 025.1 *does* offer for the per-symbol case is the `mapping ... -> rule X` mechanism described under §"Translation layer / `mapping`": the user explicitly names which rule should be inlined as a Z3 `(define-fun ...)`. That covers the small number of cases where the user knows precisely what they want translated, without committing the proposal to bulk simp-set semantics.

When the simp-set proposal lands, an anthill-side `tactic: anthill_simplify(scope: ...)` (or similar) can be added that:

1. Reads simp-marked rules from the named scope (transitive `requires` chain, namespace, named set).
2. Either applies them as anthill-side rewrites *before* SMT emission (cleaner: smaller queries) or translates them to Z3 axioms (lets Z3 do the rewriting).
3. Composes with `tactic: simplify` (Z3-native) and the rest of the catalog as expected.

Until then: write `mapping symbol -> rule X` for the specific symbols you want Z3 to know how to expand, or pre-rewrite at the anthill-rule level via `by derivation :- simp_rule_a, simp_rule_b, ...` (proposal 025) before invoking Z3.

### Translation layer

#### `logic`

SMT-LIB logic name (`"LIA"`, `"LRA"`, `"NRA"`, `"QF_LRA"`, `"AUFLIA"`, `"AUTO"`, …). Already in 025; this proposal keeps the spelling.

`AUTO` — let Z3 pick. Useful for combinator branches where each branch wants its own logic.

#### `mapping`

Already in 025. Maps anthill symbols to Z3 symbols when the auto-translation is wrong (e.g. an anthill `add` that doesn't desugar to Z3 `+`). 025.1 adds: a mapping value can be a *rule* not just a symbol, in which case the rule is inlined as the SMT-LIB definition. This unblocks user-defined arithmetic that the auto-translator doesn't recognise.

```anthill
proof distance_lower_bound
  by z3(mapping {
    distance -> rule euclidean_dist        -- inline a user-defined rule
    Vec3     -> Tuple3[Real]               -- replace the type
  })
end
```

#### `quantifier`

Choice of how to handle anthill's logic variables in goal position when emitting SMT-LIB:

| Value | Meaning |
|-------|---------|
| `forall` (default) | Universally quantify all rule-body variables |
| `skolem` | Replace existentials with fresh constants (default for "find me a counterexample" denials) |
| `none` | Emit raw asserts; caller knows quantifier handling |

Today's `emit_satisfiability_check` hardcodes `skolem` (it builds a denial expecting `unsat`). 025.1 surfaces the choice so the same emitter can produce `forall`-quantified queries for properties expecting `sat`.

### Outcome layer

#### `timeout`

Already in 025. Milliseconds. On timeout Z3 returns `unknown` — see outcome handling below.

#### `model: true` (default false)

When Z3 returns `sat`, request the model and attach it as a fact:

```anthill
fact ProofCounterexample(
  proof:    bounded_excursion_transponder,
  model:    "(model (define-fun upc () Int -7) ...)",
  variable_assignments: [
    NamedAssignment(name: "upc", value: -7),
    ...
  ]
)
```

The structured `variable_assignments` list lets downstream tooling (REPL, IDE) navigate the counterexample without re-parsing SMT-LIB.

#### `cores: true` (default false)

When Z3 returns `unsat`, request the unsat core (which assertions were necessary). Each core element is recorded against the corresponding rule:

```anthill
fact ProofCore(
  proof:           lower_violation,
  required_rules:  [step_distance_bound, comm_delay_max, GpsErrorBound]
)
```

This makes staleness tighter — a proof becomes stale only when one of its `required_rules` changes, not when *any* rule it could have used changes.

#### `interpolation: true` (default false)

For `unsat` results, request a Craig interpolant. Stored as a derived rule the user can then use. Useful for invariant synthesis.

### Outcome status

Every proof attempt records one of:

| Status | When | Effect on the obligation |
|--------|------|--------------------------|
| `proved` | every emitted query returned `unsat` | obligation discharged |
| `disproved` | any query returned `sat` for a denial (counterexample exists) | obligation is violated; load error |
| `unknown` | timeout or solver `unknown` | open obligation; staleness untouched |

Every tactic in the 025.1 catalog is **sound** — `proved` here means a real proof, not bug-finder evidence. `disproved` is a hard load error by default; downgradable to a warning via `on_disproved: warn` (use case: experimenting with weaker preconditions during proof development).

Evidence-grade strategies — bounded model checking, randomised testing — are out of scope for 025.1; see "Scope" below.

### Caching

The motivating use case: re-running `discharge.sh` (or `anthill prove <dir>`) when nothing relevant has changed should be **free** — no Z3 invocation, no SMT-LIB emission, just a cache lookup and a verdict reported. Today that re-run takes seconds at minimum and grows linearly with the number of obligations; with caching it should be ms.

#### Cache key

Z3 invocations (and the meta-tactics that wrap them) are content-addressed. The key folds in everything that could change the verdict:

```
sha256(
  emitted_smt_lib                                 -- the actual bytes Z3 sees
  || canonical_serialise(tactic_term)             -- "smt(logic: \"LRA\")"
  || hint_hash                                    -- by-derivation hints, in order
  || rule_dep_hash                                -- transitive rules consulted
  || fact_dep_hash                                -- transitive facts consulted
  || stdlib_version                               -- prelude facts + builtins
  || z3_version                                   -- solver version string
  || cache_format_version                         -- bumped on schema changes
)
```

Each component:

- **`emitted_smt_lib`** — the canonical SMT-LIB the tactic produces. For multi-query tactics (`induction`, `ranking`) this is the *concatenation* of all sub-queries in deterministic order. Whitespace/comments are stripped before hashing.
- **`canonical_serialise(tactic_term)`** — the `tactic:` argument with field-name keys sorted alphabetically and rule-references resolved to qualified names. Reordering `smt(logic: "LRA", timeout: 5000)` vs `smt(timeout: 5000, logic: "LRA")` produces the same key.
- **`hint_hash`** — for `by derivation :- rule_a, rule_b, rule_c` style proofs, the ordered list of resolved hint rule QNs. The user's question is about exactly this case: a successful proof with hints `[rule_a, rule_b, rule_c]` is cached against that hint list; a re-run with the same hints (and unchanged dependencies) hits the cache.
- **`rule_dep_hash`** — sha256 of `(qn, body_hash)` pairs for every rule transitively consulted during the proof. For a `proof` whose statement references `step_distance_bound` which references `comm_delay_max` which references `LinkParameters` and `KinematicAssumptions`, the rule-dep hash covers all of them. Implementation: compute once during proof emission via the existing `requires_chain` traversal (rustland/anthill-core/src/kb/mod.rs).
- **`fact_dep_hash`** — sha256 of `(functor, ordered_fields)` for every fact whose head matches a body atom. Editing `fact GpsErrorBound(epsilon: 0.1)` to `0.05` invalidates every proof that consulted `GpsErrorBound`.
- **`stdlib_version`** — the prelude's git rev (or content hash if running uncommitted). Editing `stdlib/anthill/prelude/numeric.anthill` could change `add` semantics; cache must invalidate.
- **`z3_version`** — `z3 --version` output, captured at cache-write time. Different solver versions can give different verdicts (especially `unknown`). Each solver subtree composes its key from the *appropriate* version string — `dreal/` entries fold in `dreal --version`, `cvc5/` entries fold in `cvc5 --version`. The solver identity itself is not in the key (the directory encodes it); only its version is.
- **`cache_format_version`** — fixed integer per solver subtree; bumped when *that solver's* cache value layout changes. Old entries become misses, not corrupted reads. A Z3-side schema bump leaves the dReal subtree untouched.

#### Cache value

```
{
  "key":             "<sha256-hex>",
  "verdict":         "proved" | "disproved" | "unknown",
  "model":           "(model ...)" | null,         -- only on disproved or model: true
  "unsat_core":      ["rule_qn_a", ...] | null,    -- only on proved + cores: true
  "interpolant":     "(...)" | null,               -- only on proved + interpolation: true
  "wall_clock_ms":   123,                          -- telemetry
  "z3_version":      "4.13.4",
  "tactic_summary":  "smt(logic: \"LRA\")",        -- human-readable, for debugging
  "proof_qn":        "anthill.examples.lf1.safety.gps.lower_violation",
  "stamp":           "2026-04-27T10:13:42Z"
}
```

The `proof_qn` and `tactic_summary` fields are *not* part of the key — they're for human inspection (`anthill prove --show-cache`). The key alone determines hit/miss.

#### On-disk layout

```
<cache_root>/projects/<sha256(repo_root)>/proofs/
  z3/                                   -- solver-specific subtree
    v1/                                 -- cache_format_version (per-solver)
      <first-2-hex-of-key>/
        <full-key>.json
  dreal/                                -- future: independent subtree
    v1/
      ...
  cvc5/                                 -- future
    v1/
      ...
```

Where `<cache_root>` follows the XDG Base Directory spec:

| OS | Default `<cache_root>` |
|----|------------------------|
| Linux | `$XDG_CACHE_HOME/anthill/` (default `~/.cache/anthill/`) |
| macOS | `~/Library/Caches/anthill/` |
| Windows | `%LOCALAPPDATA%/anthill/cache/` |

The `projects/<sha256(repo_root)>/` segment namespaces the cache by project, so `~/.cache/anthill/` can hold cache trees for many anthill repos without collisions. The hash is over the repo root path (the parent of `.git/`), captured at proof-time. (Renaming or moving a repo cleanly invalidates its cache; that's correct — the project identity changed.)

Solver name is a *top-level* directory, not a key field. Rationale:

- A Z3 version bump changes `z3_version` in the key and invalidates Z3 entries; dReal entries are untouched and still hit. Putting them under one shared cache would force a global key bump.
- Cache value formats differ per solver (Z3's `(model ...)` text vs dReal's box assignment vs CVC5's; different unsat-core formats). A solver-aware reader knows which format to expect from the path; a flat cache would need a discriminator field.
- `cache_format_version` is per-solver — bumping the Z3 entry layout doesn't churn dReal cache entries.
- Operational commands (`--gc-cache --solver z3`) are natural; without a `--solver` flag they walk every subtree.

Inside the solver subtree, two-level hex fanout (`<first-2-hex>/<full-key>.json`) keeps directory listings tractable on filesystems that struggle past ~10k entries.

The cache key itself does *not* include the solver name — the directory encodes it. This means the same logical proof discharged by two different solvers gets two cache entries (one in `z3/`, one in `dreal/`), which is the correct semantics: the verdicts are independent.

#### Cache location: per-user, XDG-compliant, *outside* the repo

Default: `<cache_root>/projects/<sha256(repo_root)>/proofs/` per the XDG Base Directory spec — i.e. `~/.cache/anthill/...` on Linux, `~/Library/Caches/anthill/...` on macOS. Cache files live entirely outside the working tree, so:

- no `.gitignore` entry needed (nothing for git to see),
- no merge conflicts on cache JSON,
- no risk of someone committing a hand-edited `"verdict": "proved"` claim,
- no repo bloat from accumulated cache entries.

Earlier drafts of this proposal suggested an in-repo path (`<repo>/.anthill/proofs/cache/`); this was wrong because it forced every project to carry a `.gitignore` entry it wouldn't otherwise need, and a missed entry would silently start committing cache JSON.

XDG is the right shape because the cache is a function of `(repo identity, spec content, solver version)`, all of which are reproducible — keeping it in the per-user cache directory matches `cargo`'s incremental cache, `sccache`, Bazel's repository cache, mypy's `.mypy_cache`-when-`MYPY_CACHE_DIR`-is-set, and so on. The user's broader expectations about "where build artifacts go" are met by following the OS convention.

Override knobs (in priority order, highest first):

1. CLI flag: `anthill prove --cache-dir <path>`
2. Environment variable: `ANTHILL_CACHE_DIR=<path>`
3. Project config: `anthill.toml` `proof.cache.dir = "..."`
4. Default: XDG (`<cache_root>/anthill/projects/<repo-hash>/proofs/`)

Per-solver subtree structure is preserved relative to whichever root wins.

CI gets warm caches via the *platform's* cache mechanism, pointing at whichever location the override-chain resolves to. Easiest pattern: set `ANTHILL_CACHE_DIR` to a workspace-relative path so `actions/cache` knows where to find it.

```yaml
# GitHub Actions: park the cache in the workspace and persist across runs.
- name: Configure anthill cache
  run: echo "ANTHILL_CACHE_DIR=${{ github.workspace }}/.cache/anthill" >> $GITHUB_ENV

- uses: actions/cache@v4
  with:
    path: .cache/anthill
    key: anthill-proofs-${{ hashFiles('**/*.anthill', 'stdlib/**') }}-${{ runner.os }}-z3-${{ env.Z3_VERSION }}

- run: anthill prove .
```

The pattern is the same on GitLab CI, Buildkite, CircleCI — each has a native cache primitive keyed on a content hash. The PR experience is "fast when nothing relevant moved" without any cache files in `git log`.

Other location choices and why they lose:

| Location | Issue |
|----------|-------|
| `<repo>/.anthill/proofs/cache/` (in-repo, gitignored) | requires every project to add a `.gitignore` line; missed line silently commits cache JSON |
| `<repo>/.anthill/proofs/cache/` (in-repo, committed) | trust (unsigned JSON) + merge conflicts + repo bloat |
| `target/anthill-cache/` | `target/` is conventionally Rust-specific; the anthill toolchain is polyglot |
| `~/.anthill/cache/` (flat per-user, no project namespacing) | collides across projects with the same proof name |

`anthill prove --gc-cache` walks all subtrees of the resolved cache root by default; `--scope project` restricts to the current repo's namespace.

#### Lookup, hit, miss, write

```
on each proof obligation:
  1. compute cache key (cheap: hashing, no Z3)
  2. look up <repo>/.anthill/proofs/cache/<key>.json
     hit:  verdict was previously "proved"   → report "✓ proved (cached)"
           verdict was previously "disproved" → load error (still a counterexample)
           verdict was previously "unknown"   → respect cache_unknown_ttl, see below
     miss: emit SMT-LIB, run Z3, write result, report verdict
  3. summary line includes cache hit/miss counts
```

Cached `disproved` entries still produce a load error (the counterexample is still real). Re-running with `--no-cache` allows debugging — useful when fiddling with assumptions.

Cached `unknown` entries are subject to a TTL (`cache_unknown_ttl`, default 24h). After expiry the entry is treated as a miss and Z3 is re-invoked. This avoids permanent timeout-induced blocks when Z3 is updated or the user fiddles with `timeout: ...`.

#### Invalidation

Invalidation is purely *implicit* via the cache key. Any change that touches a contributing rule, fact, prelude file, or solver version produces a different key; the previous entry is simply unreferenced (not deleted). Stale entries accumulate over time; `anthill prove --gc-cache` walks the cache directory and deletes entries with no live key.

The existing `trust: stale` metadata (025) does not gate the cache — they're orthogonal mechanisms. `trust: stale` is about *human review of the proof's continued relevance*; cache validity is about *bit-identical inputs producing the previously-recorded output*. A proof can be `trust: stale` and have a valid cache entry simultaneously (the user knows the proof needs re-thinking but the math still holds for what it does prove).

#### Operational commands

```
anthill prove <dir>                  -- normal: cache hits silent, misses run Z3
anthill prove <dir> --no-cache       -- ignore cache; force re-run; still write entries
anthill prove <dir> --refresh-cache  -- ignore cache reads; rewrite entries
anthill prove <dir> --show-cache     -- list cache entries with proof_qn / tactic / verdict
anthill prove <dir> --gc-cache       -- delete entries with no live key
anthill prove <dir> --stats          -- cache hit-rate + Z3 wall-clock total
```

CI uses `anthill prove` (default cache behaviour). Local development uses `--refresh-cache` after Z3 upgrades or stdlib changes that the dependency hash didn't catch (escape hatch for hash-computation bugs).

#### What gets cached, what doesn't

- Cached: `proved`, `disproved`, `unknown` from any tactic that completes.
- Not cached: solver crashes, OOM, kernel signals — these write nothing and the next run retries.
- Not cached: tactics that explicitly opt out via `cache: false` (rare; mainly for tactics whose output is non-deterministic — none in 025.1's catalog, but the field exists for future use).

#### Relationship to solver-internal caches

The anthill proof cache described above is *external* to whatever caching the solver itself does. Z3 has internal incremental-solving (push/pop), expression hash-consing, and AST sharing; dReal has interval-propagation memoisation; CVC5 has its own. Those are per-process and unaffected by ours. Our cache lives one level up: between the anthill front-end and the solver invocation. A cache hit means "we never spawn the solver"; the solver's internal caches don't enter the picture.

This is also why the per-solver subtree (`cache/z3/`, `cache/dreal/`, …) is the right structural split — each solver's external cache is independent in the same way each solver's internal cache is, and the layouts can evolve at different rates.

#### Concrete savings on lf1

Local development, no-change re-run:

- Today's `discharge.sh`: 5 obligations × ~200ms Z3 spawn = ~1s. Plus parser load + SMT emission = ~3s wall-clock.
- With caching: 5 cache lookups × ~1ms = ~5ms. Plus parser load = ~500ms wall-clock.
- ~6× speedup on lf1, more on bigger spec sets.

CI, between PRs that don't touch proofs:

- Today: full re-discharge every run.
- With caching + `actions/cache@v4`: cache restore (~50ms) + cache lookups (~5ms total) + parser load. The proof step goes from "always slow" to "slow only when contributing inputs actually moved."

This subsumes today's "rerun `discharge.sh` on every change" pattern — an unchanged proof is free locally, and CI surfaces only the proofs whose contributing inputs actually moved.

### Worked example: lifting the lf1 ranking-function discharge

The current state (after the manual discharge in `lf1_real_spec_test.rs`):

```anthill
-- safety_transponder.anthill (today)
fact PostArmedExcursionBound(max_ticks: 6)        -- hand-plugged meta-theorem

rule excursion_distance_bound(?excursion)
  :- transponder_step_distance_bound(?delta_t),
     PostArmedExcursionBound(max_ticks: ?n),
     ?excursion = ?n * ?delta_t

proof bounded_excursion_lower_violation_transponder
  by z3(logic: "LRA")
end
```

The hand-plugged fact disappears once 025.1 lands:

```anthill
-- safety_transponder.anthill (with 025.1)

-- Ranking function on TransponderFollowerState.
rule neg_upc(?state, ?r)
  :- ?state = TransponderFollowerState(upc: ?u, _),
     ?r = neg(?u)

-- The "bad step" predicate: post-armed transition where distance grew.
rule bad_step(?state, ?reading, ?next_state)
  :- step(?state, ?reading) = ?next_state,
     ?state.prev_distance = some(?prev_d),
     gt(?reading.distance, ?prev_d)

-- The meta-theorem that used to be hand-plugged.
proof post_armed_excursion_bound
  by z3(tactic: ranking(
    measure:         neg_upc,
    decrease_when:   bad_step,
    state_sort:      TransponderFollowerState,
    well_founded_on: Int,
    logic:           "LIA"
  ))
end

-- excursion_distance_bound now reads bound_witness from the proved fact:
rule excursion_distance_bound(?excursion)
  :- transponder_step_distance_bound(?delta_t),
     RankingProof(name: post_armed_excursion_bound, bound_witness: ?n),
     ?excursion = ?n * ?delta_t

-- The downstream LRA proof is unchanged — it just reads the proved bound
-- instead of the axiomatised one.
proof bounded_excursion_lower_violation_transponder
  by z3(logic: "LRA")
end
```

The four SMT sub-queries from the manual test (`anthill-smt-gen/tests/lf1_real_spec_test.rs::lf1_transponder_excursion_ranking_function_manual`) are now emitted automatically:

- query 1 (`boundedness`) ← `ranking` tactic, dual query 1
- query 2 (`decrease`) ← `ranking` tactic, dual query 2
- query 3 (`bounded-excursion ⇒ safety`) ← downstream `bounded_excursion_*_violation_transponder` proofs (already discharged)
- query 4 (`sanity`) ← becomes a separate `proof excursion_bound_non_vacuity by z3(tactic: smt(logic: "LRA"))` if we want to keep it

The manual test stays in place as a regression check until the 025.1 implementation reproduces the same verdicts; then it can be deleted.

## Implementation sketch

The work splits across `anthill-smt-gen` and `anthill-core`:

`anthill-smt-gen`:
1. Parse the tactic term out of the proof's `by z3(...)` arguments. Today the args go straight into a flat `BTreeMap<String, Value>`; extend to recognise tactic-shaped values.
2. For each tactic variant, an `emit_*` function that produces one or more SMT-LIB strings. The current `emit_satisfiability_check` becomes the body of `emit_smt`.
3. Combinator dispatch (`then`, `or_else`, `repeat`) is recursive — each combinator runs its sub-tactics and combines verdicts.
4. Cache layer: hash → verdict, persisted to `target/anthill-smt-cache/` (paths configurable).

`anthill-core`:
1. Proof-loading pass reads `RankingProof` / `ProofCounterexample` / `ProofCore` facts back into the KB so downstream rules can pattern-match on them.
2. `trust: stale` propagation reads `ProofCore.required_rules` (when present) instead of conservatively traversing the full requires chain.

Tactics are open-set: a tactic name that the implementation doesn't recognise is a load error, not silently ignored. New tactics ship by extending the emitter, not the grammar.

## Grammar

Extends the `field_value` admissible-value grammar from 025; no new top-level form, no changes to `proof_declaration`.

```
field_value ::= literal                              -- as in 025
              | name                                 -- as in 025
              | tactic_term                          -- new in 025.1

tactic_term ::= name                                 -- bare: `smt`, `simplify`
              | name '(' tactic_args ')'             -- applied: `smt(logic: "LRA")`
              | combinator '(' tactic_args ')'       -- `then(...)`, `or_else(...)`
              | 'raw' '(' string_literal ')'         -- escape hatch to native Z3 syntax

combinator ::= 'then' | 'or_else' | 'repeat' | 'par'

tactic_args ::= field_list                           -- nested k:v pairs
              | tactic_term (',' tactic_term)*       -- positional sub-tactics (combinators)
              | field_list ',' tactic_term (',' tactic_term)*
                                                     -- mixed (e.g. repeat(times: 5, sub_tactic))

mapping_entry ::= name '->' name                     -- as in 025
                | name '->' 'rule' name              -- new in 025.1: inline a rule
```

Tactic terms reuse the existing field-list grammar — no new top-level form to parse.

## Scope: what 025.1 is and is not

**Is**:
- A first-class place to write tactic strategies inside `by z3(...)`.
- A path to lift the lf1 ranking-function meta-theorem into the language.
- A schema for proof outcomes (counterexamples, cores) as KB facts.
- Sound-only: every tactic in the catalog yields `proved` on success or fails. No bug-finder-grade results are returned dressed up as proofs.

**Is not**:
- A full LCF-style tactic language. Tactics are a closed catalog; user-defined tactics are out of scope. (User-defined *measures* and *predicates* are in scope — they're just rules.)
- A non-Z3 solver story. dReal / CVC5 / Vampire would each get their own `by <solver>(...)` form. Some args (`timeout`, `logic`) overlap — that's intentional, not a leaky abstraction. The solver's idiosyncratic knobs stay in its own namespace.
- A replacement for `by derivation`. SLD-trace proofs and SMT-discharged proofs solve different problems; both stay first-class.
- A home for evidence-grade strategies (BMC, randomised testing, statistical model checking). These genuinely need a soundness-distinction mechanism, and that question warrants its own proposal rather than being grafted onto the tactic DSL. Until that proposal lands, `by test(runs: ...)` from 025 remains the only evidence-grade strategy and it stays under `proof` per 025.

  **Possible extension (leading candidate)**: keep `proof` as the only top-level keyword and add an evidence-level *inside* the proof expression rather than introducing a sibling form. Concretely something along the lines of:

  ```anthill
  proof transponder_no_collision_within_100_ticks
    evidence                                          -- declarative level inside the body
    by z3(tactic: bmc(transition: step, depth: 100, ...))
  end
  ```

  The `evidence` line is part of `proof_body` (alongside `:- proof_term` and `query "..."`), not a new top-level keyword. The reader still sees the strength in the text — it's right under the proof name — but the grammar grows by one keyword in body position rather than at the top level. The recorded fact gets a `level: evidence` tag instead of a separate `EvidenceResult` type. Detail design (where exactly in `proof_body` it sits, default level for missing annotation, deprecation path for `by test(...)`) is the dedicated proposal's job; mentioning it here so the deferred work has a concrete starting shape.

## Open questions

1. **Inferring `state_sort` and `well_founded_on` for `ranking`.** Today both are required arguments. They're recoverable from the `measure` rule's signature — could default to "the sort of the first parameter" / "the codomain." Worth doing if it doesn't cost diagnostic clarity.
2. **`induction` over user-defined sorts.** The auto-generated induction rule is fine for `enum` and structural recursion, but a generic well-founded induction (over `compare`-ordered carriers) needs a `WellFoundedOrder` provides-fact. Defer to a follow-up.
3. **Counterexample replay.** When Z3 returns `sat`, the assignment is in SMT-LIB types (`Int`, `Real`, `Bool`). Reconstructing an anthill `Term` from that is straightforward for ground Int/Float/String/Bool but unclear for sort-typed values. The generated `ProofCounterexample` fact stores the raw model string as a fallback while the structured form is best-effort.
4. **Streaming verdicts for combinators.** `or_else` should short-circuit on the first success; `then` should stop on the first failure. Implementation needs a `Verdict::Fail(reason)` enum richer than today's binary `Result<bool, _>`.
5. **Whether `mapping ... -> rule X` belongs here or in 025.** It started as a one-line addition to `mapping`'s value grammar, but it's semantically heavier than the symbol-renaming case 025 covers. May warrant pulling out to its own micro-proposal.
6. **`chc` tactic — separate proposal?** Translating anthill safety obligations to HORN logic for Spacer is a real chunk of work (encoding the transition relation, mapping anthill sorts to Spacer-acceptable types, parsing Spacer's invariant certificates). Worth a sibling proposal once `ranking` is landed and we have a calibration point for what it leaves undone.
7. **Should the catalog grow by adding to the implementation, or by users writing `raw(...)` and us promoting popular incantations?** The latter is the Z3 community's de-facto path for tactic discovery; we may want to tail it rather than guess what users need.
8. **Evidence-grade strategies** — bounded model checking, randomised testing, statistical model checking — need a soundness-distinction mechanism. 025.1 deliberately punts on this so the tactic DSL doesn't have to settle the question. The leading candidate (per "Scope" above) is an in-`proof` annotation rather than a new top-level keyword: an `evidence` line inside the proof body alongside `:- ...` and `query "..."`. Once that lands, BMC slots into the existing tactic-term grammar without further changes; only the surrounding annotation moves.
9. **Simp-set semantics** — see §"Anthill-rule-aware simplification (deferred)". Marking which equational rules are intended as rewrites needs its own proposal (annotation form, scope, default simp set, interaction with `requires`). Until then, `tactic: simplify` is Z3-native only and `mapping ... -> rule X` is the per-symbol workaround.

## Summary

`by z3(...)` becomes a tactic DSL via a `tactic:` argument that takes structured tactic terms. The DSL is **thin** wherever Z3 already has the construct: combinators (`then`/`or_else`/`repeat`/`par`), `smt`, `simplify`, `logic`, `timeout`, `model`, `cores`, and a `raw("...")` escape hatch all serialise straight to Z3 input. The DSL is **anthill-layer** only where the work involves multi-query assembly or KB-fact emission: `induction` (uses anthill's inductive-sort definitions to generate case-split + IH subgoals) and `ranking` (assembles the dual boundedness + decrease queries; emits `RankingProof`). A future `chc` tactic — Spacer-mediated safety-property verification — is the architectural sibling that should subsume parts of `ranking` for transition-system shapes and supply BMC as a side effect.

The catalog is **sound-only**: every tactic yields `proved` on success or fails. Evidence-grade strategies (BMC, randomised testing, statistical model checking) are deferred to a future proposal that gets to design the soundness-distinction mechanism — surface keyword, modifier, or fact tag — on its own merits. Until then, `by test(runs: ...)` from 025 remains the only evidence-grade strategy.

Three argument layers — translation, tactic, outcome — organise the catalog without enforcing groups syntactically. Proof results land back in the KB as facts (`RankingProof`, `ProofCounterexample`, `ProofCore`) so downstream rules can read them and staleness can be precise.

The lf1 ranking-function meta-theorem — currently hand-plugged as `PostArmedExcursionBound(max_ticks: 6)` and proved off-line in `lf1_real_spec_test.rs` — becomes the first user of the `ranking` tactic and the worked example throughout this document.
