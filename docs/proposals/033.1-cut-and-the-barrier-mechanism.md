# Proposal 033.1: Cut and the barrier mechanism

**Status:** Implemented (WI-568)
**Depends on:** [033-resolver-primitives-and-disjunction](033-resolver-primitives-and-disjunction.md)
**Affects:** `rustland/anthill-core/src/kb/{resolve,mod,load}.rs`, `rustland/anthill-core/src/parse/convert.rs`, `tree-sitter-anthill/grammar.js`, `stdlib/anthill/kernel/kernel.anthill`

## Motivation

Proposal 033 deferred cut to a child proposal (033 §Future work):

> Cut commits to the *current rule invocation* — given `a(?x) :- b(?x), !, c(?x)`,
> when `!` fires it must discard (i) any other clauses of `a` still queued at the
> parent `ChoicePoint`, and (ii) every `ChoicePoint` created during the
> resolution of `b(?x)`. Choice points created *after* `!` (during `c(?x)`) are
> untouched.

`push_choice` (033 / WI-075) *creates* choice points; cut *prunes* them back to a
barrier. The two are inverse operations on the same `ChoicePoint` stack, and cut
is the eventual clean form for any "first match wins, else default" priority that
is otherwise written with a negation-as-failure existence wrapper.

## Scope correction (load-bearing)

Proposal 033 Requirement 3 claimed cut is expressible as rewrites in
`anthill.kernel` *without core changes*. That is over-optimistic and contradicted
by 033's own §Future work. `push_choice` creates choice points; cut prunes them
back to a barrier — the inverse — which genuinely requires the two core resolver
additions below. The no-core-changes claim holds only for the **derived**
combinators (`once`, `if-then-else`, `findall`), each a pure rewrite over cut.

## Design

### Surface syntax

`!` is a body goal (a conjunct), parsed as a dedicated `cut` node. It is distinct
from the prefix `!`/`not` operator (`! <atom>`, negation): a `!` with no following
operand is cut, a `!` applied to an atom is negation. The two overlap on the
leading `!` token and are declared as a GLR conflict (`[$.cut, $._prefix_op]`),
resolved by lookahead — only the production whose continuation parses survives.
`!` is rejected loudly in rule-head position (a head is a conclusion, not a
control goal), mirroring the goal-position `let` rejection (proposal 049).

The converter lowers `!` to a nullary `cut` goal; the short name resolves to
`anthill.kernel.cut` via the implicit kernel vocabulary, so no import is needed.
`anthill.kernel.cut` is declared `operation cut() -> Bool` — the nullary
user-facing face; its resolver-internal barrier argument (below) is not written.

### The barrier mechanism

Three pieces, mirroring 033 §Future work:

1. **Per-invocation barrier.** When a rule body containing a `!` is opened
   (`with_fresh_vars` / `step_choice_point`), a fresh monotonic `BarrierId` is
   allocated — `BarrierId` is to cut what `VarId` is to a query variable: a fresh,
   per-invocation tag. The choice point that selected the rule (the *call frame*
   `F0`) is tagged with it (`FrameState::ChoicePoint::cut_barrier`).

2. **`cut(B)` opening.** The `!` markers in the opened body are *baked* to
   `cut(IntLit(B))`, parallel to De Bruijn-var opening: the concrete barrier is
   substituted in, so the cut goal carries `B` wherever it later flows down the
   resolution spine (a body's continuation goals are appended to the child frame's
   goal list, so a `!` written in `a`'s body may fire several frames deeper —
   baking is what keeps it bound to `a`'s invocation).

3. **`apply_cut(B)`.** When `cut(B)` fires, the resolver finds the frame tagged
   with `B` (that is `F0`) and neutralizes it and **every choice point stacked
   above it** — `next` advanced past all candidates, the delay-fallback cleared —
   so backtracking pops straight through them. The current (top) frame, where the
   cut fired, is left intact to continue with the body's tail.

Finding the barrier frame by scan, then neutralizing the contiguous suffix above
it, is the WAM "reset B to the cut barrier" reset: the explicit choice-point stack
*is* the call-tree spine, so "stacked above `F0`" is exactly "created during this
invocation's body". Requirement (i) (other clauses of the cut's rule) is `F0`'s
own remaining candidates; requirement (ii) (choice points created resolving the
goals before the `!`) is the suffix above `F0`.

### Disjunction–cut interaction falls out

- **Outer cut prunes an inner `or` (transparency).** An `or(...)`'s `push_choice`
  continuations are pushed *above* the enclosing rule's `F0`, so a cut in that
  rule's body neutralizes them along with everything else above `F0`.

- **An inner cut is opaque to the outer `or`.** A cut inside an inner rule (or
  inside `or`'s own body) carries that inner invocation's barrier and reaches down
  only to *its* `F0`, which sits above the outer `or`'s choice point — the outer
  `or`'s alternatives are below the inner barrier frame and are preserved.

This is Prolog's "disjunction transparent to outer cut, opaque to inner cut",
achieved with no special case.

### Interaction with NAF and delay

- **Cut under `not(P)`.** NAF runs `P` in a *sub-stream* (a fresh `SearchStream`).
  A cut in `P` allocates its barrier from, and prunes, the sub-stream's stack —
  scoped to the negation's proof, never the outer search.

- **Cut with delay / residualization.** The cut goal never delays (it always
  succeeds and commits). A guard *before* a cut that delays rotates as usual; the
  cut then fires on the rotation, and a guard that ultimately flounders
  residualizes honestly — the spent barrier on an already-popped frame is inert.

## Derived combinators (no further core)

Once cut lands, these are pure rewrites in `anthill.kernel`:

- `once(g)` ≡ `(g, !)` lifted into a synthesized rule.
- `if-then-else(c, t, e)` ≡ `(c, !, t) ; e`.
- `findall` ≡ meta-call plus collection.

(Not built here — listed as the follow-on surface these primitives enable.)

## Non-goals / limitations

- **Cut nested inside an argument term** (e.g. a hand-written `or((g, !), h)`,
  with `!` *inside* the term passed to `or` rather than as a conjunct) is not
  baked. The supported surface form is the conjunct `!`, which is what `once` /
  if-then-else and priority rules expand to.

- **scaland parity** (grammar + loader; resolver port if scaland resolves) is a
  follow-up.

## Implementation notes

- `BuiltinTag::Cut`, special-cased in `step_init` alongside `Not` / `HoApply` /
  `PushChoice` (its effect is on the choice-point stack, not on σ).
- `FrameState::ChoicePoint` gains `cut_barrier: Option<i64>`; `SearchStream` gains
  a monotonic `next_barrier` allocator (local to the stream, so NAF sub-streams
  are isolated).
- Baking detects a top-level cut marker among the opened body occurrences via
  `get_builtin_view` (identical to how the goal is later classified) and swaps it
  for a `Value::Term(cut(IntLit(B)))`.

## Immediate consumer (non-blocking)

`anthill.realization` type-mapping priority ("specific key wins, else default",
WI-089) is the motivating clean-form target — `TypeMapping(...), !, default`
instead of an NAF existence wrapper. NOTE: the delivered WI-089 implementation
encodes that priority *caller-side in Rust* (`resolve_type_mapping`), reading the
keyed facts through `kb.query_view` (a discrimination-tree match, **not** SLD), so
cut — an SLD choice-point operation — does not reach it. Adopting the cut form
there would require moving the priority into an in-language rule resolved via
`kb.resolve`, which the cpp-gen `&KnowledgeBase` borrow does not currently permit.
Recorded here so the cross-reference is not mistaken for a wiring that exists.
