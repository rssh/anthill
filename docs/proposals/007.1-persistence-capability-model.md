# Proposal 007.1: Persistence Capability Model — Corrected

**Status:** Draft
**Revises:** 007 (Pluggable Persistence Layer) — §2 (Store / StoreCaps), §4 (Core Operations)
**Relates to:** 053 (per-functor fact monotonicity — the write side), 036 (domain store sorts)

## Why this revision

007 §2 proposes a store's capability as a returned value:

```anthill
sort StoreCaps { entity queryable  entity bulk }
operation caps(store: Store) -> StoreCaps
rule caps(FileStore(?)) = bulk
rule caps(SqlStore(?))  = queryable
```

Two things about that model did not hold. This revision records the corrected one.

## Correction 1 — capability is a subsort, not a returned value

`caps` / `StoreCaps` was **never implemented**. The stdlib
(`stdlib/anthill/persistence/store.anthill`) realizes the same queryable/bulk
distinction **structurally, as a sort hierarchy**:

```anthill
sort Store          { persist, retract, flush }          -- base
sort QueryableStore { fact Store   operation retrieve }  -- "queryable"
sort BulkStore      { fact Store   operation pull }      -- "bulk"
```

A backend states its capability by **which subsort it satisfies**
(`fact QueryableStore[SqlStore]`, `fact BulkStore[FileStore]`), not by a
`caps(store)` rule. This is the better encoding: a store's capability *is* the
set of operations it provides, so the sort that carries those operations is the
capability — the backward-chaining engine (kernel §8.3) dispatches on subsort
membership, and a store that provides neither `retrieve` nor `pull` is simply a
bare `Store`. **`caps` / `StoreCaps` is retired;** read 007 §2 and §5 through the
`QueryableStore` / `BulkStore` subsorts.

## Correction 2 — there is no *write*-capability classifier, by design

007's `caps` is about the **read / retrieval** model — how backward chaining
gets facts *out* of a store (delegate to the backend vs. load all into memory).
It says nothing about **write** ability, and 007 §4 ("Core Operations") is only
the `persist` / `retract` / `pull` / `flush` *signatures*, not a capability.

It is tempting to add a symmetric *write*-capability lattice — roughly
`none ⊏ persist-only (append-only sink) ⊏ persist+retract (mutable)`, a read-only
view as `none` — and classify each store into a tier (an `AppendOnlyStore` /
`MutableStore` subsort pair over `Store`). **We reject that.** In any nontrivial
store the write policy is store-dependent logic that no fixed tier captures: a
SQL store retracts a table via `DELETE` but not a materialized view, gated by
grants and triggers; an API-backed store permits delete for some resource types
and `405`s others; a tiered store is mutable while hot and append-only once
archived. A coarse per-store tier is a lie about a per-predicate, often
runtime-only, reality.

So write policy is **not a store-level capability** at all. It is a
**per-predicate property the owning store provides** — see 053:
`fact_monotonicity(functor) -> Monotonicity`, answered for an in-memory functor
by reflect rules and for an external-store functor by the owning store's own API
(the store is the single authority; there is no separate `monotonicity ≤
capability` bound to check it against). A backend that cannot honor a declared
`non_monotone` retract fails **loudly at the write** via the `Error` effect
(037), where the store-specific logic actually lives — not at a static load-time
capability check.

## Net

- **Read capability** — the `QueryableStore` / `BulkStore` subsorts of `Store`
  (as implemented). `caps` / `StoreCaps` from 007 §2 is retired.
- **Write capability** — not a store classifier. Per-predicate, store-provided
  via `fact_monotonicity` (053), enforced at the write via `Error`, not a static
  bound.
