# Proposal 027.1: Allocator Effects via `Modify[result]` and Local Discharge

**Status:** Draft
**Depends on:** [027-effect-handlers-and-standard-effects](027-effect-handlers-and-standard-effects.md) (handler model + standard effect catalog), [037-anthill-state-model](037-anthill-state-model.md) (Cell, allocator operations), [041-operation-result-naming](041-operation-result-naming.md) (one-arm typer extension making `result` resolvable in effects rows; WI-261)
**Related:** [039-term-level-constants](039-term-level-constants.md) (const validator inherits the dishonesty if allocators stay pure), [026-expression-evaluator](026-expression-evaluator.md) (CSE / inlining / folding need an honest effect declaration)
**Affects:** `stdlib/anthill/prelude/cell.anthill` + `map.anthill` + `substitution.anthill` (allocator signatures), 037 §2 and §Identity (one-line cross-reference), typer (current `is_local_resource` filter at `rustland/anthill-core/src/kb/typing.rs:270–307` does symbol-name lookup for parameter/let-binding discharge; this proposal adds return-flow escape analysis: substitute `result → x` at call sites binding an allocator's return, then check whether `x` flows out of the operation's return expression — lift to `Modify[result]` when it does)

## Motivation

Proposal 037 §2 declares allocator operations as effect-pure:

> Construction is allocation, not mutation — it doesn't modify any existing Cell, so it doesn't carry `Modify[anything]`. […] `Map.empty()`, `Substitution.empty()`, `List.nil()` are all effect-pure even though each call yields a fresh handle.

And 037 §Identity says:

> Two calls to `Cell.new(0)` produce two distinct cells even with identical initial values. Identity is allocation-time, not value-time.

Jointly: an operation declared with no effects whose runtime semantics break equational reasoning. This stance — *pure-but-identity-distinguishing* — is rare in the algebraic-effects literature. Most systems either declare the operation effectful (Haskell IO/ST, Koka `heap`, Frank / Eff handler-defined) or accept the impurity without an effect system (OCaml). 037 took a fourth path: pragmatically defensible (keeps allocator call sites lightweight), but it leaves the effect system materially less useful than it should be.

Two pressures push the revision now:

1. **039's const validator needs honest input.** A `const` body must be "foldable and effect-free." If `Cell.new` is declared with no effects, the validator admits `const C: Cell[Int] = Cell.new(0)` — accidentally producing a singleton-cell semantics that depends on a fragile invariant the user has to know. The right answer is for `Cell.new`'s declaration to forbid this admission outright.

2. **026's evaluator transformations will mis-rewrite.** As load-time folding, CSE, and inlining mature (and especially as 026 M5+ adds Suspension / Branch state handling), the effect annotation becomes load-bearing. Today's "pure-but-distinguishing" allocator lets a CSE pass collapse `(Cell.new(0), Cell.new(0))` to `let x = Cell.new(0); (x, x)` — silently wrong.

And a third pressure becomes obvious once you start counting allocators in real code:

3. **Pollution.** Almost every nontrivial operation allocates a transient Cell, Map, or Substitution. A naive "every allocator carries its own effect kind" stance would put `Alloc[...]` lines in the effect row of nearly every signature in the codebase. Caller signatures fill with allocations they don't care about and can't observe. The effect system stops being read.

This proposal addresses all three by unifying allocation into the existing `Modify[T]` machinery and adding a **local-body-flow discharge rule** that already exists for `Modify[c]` on input parameters — extended to cover `Modify[result]` on returns.

## Design

### Allocator effects collapse into `Modify[result]`

There is no new `Alloc[T]` effect kind. Allocator operations declare:

```anthill
operation new(initial: V) -> Cell[V] effects Modify[result]
operation set(c: Cell[V], v: V) -> Unit effects Modify[c]
operation get(c: Cell[V]) -> V                                -- pure
```

`result` is the reserved name for the operation's return slot, introduced by [proposal 041](041-operation-result-naming.md) and already reserved for `ensures` clauses in the kernel spec (§5.4).

**Why this unifies cleanly.** Allocation isn't a fundamentally distinct kind of effect — it's the introduction of state with a fresh identity. Setting `c` to `v` mutates an existing target whose identity is shared with the caller; constructing a fresh `Cell` mutates a target whose identity is, momentarily, not shared with anyone. The mechanism is the same; only the target source differs:

| Effect occurrence | Target source | Identity behavior |
|---|---|---|
| `Modify[c]` on input parameter `c` | Caller-provided | Caller observes the modification |
| `Modify[result]` on return | Fresh, owned by callee until returned | Caller observes only if they bind the result |

The 037 lines that argue "allocation isn't mutation" remain correct in their narrow sense — allocator ops don't mutate any *pre-existing* cell. But they do introduce mutable state, and that's enough to break referential transparency, which is what an effect declaration captures.

### Discharge rule — local body flow, same as `Modify[c]`

The typer already does the analysis needed. For `Modify[c]` where `c` is an input parameter, the propagation rule is: if the body invokes `Modify` on a value that traces back to `c` along the body's data flow, the operation's signature must declare `Modify[c]`; the caller substitutes the actual argument for `c` at the call site.

**Extension for `Modify[result]`:** treat `result` as an additional bind site whose flow originates at the return expression. The discharge rule is the dual:

> An operation that calls a callee with `Modify[result]` in its row inherits `Modify[x]` for the local name `x` it binds the result to. If `x` does not escape the operation's body, the effect is discharged at the function boundary.

"Escape" is determined by the same local flow analysis already used for Modify-on-parameter:

- Returned in the operation's return expression → escapes → propagates.
- Stored in a fresh value that itself escapes → escapes → propagates.
- Bound by a `let` and only consumed by other operations whose effects don't re-export it → does not escape → discharged.
- Passed to a closure that captures and returns it → escapes → propagates.

This extends the typer's existing parameter-discharge filter (`is_local_resource` at `rustland/anthill-core/src/kb/typing.rs:270–307`, which today checks symbol-name membership in the body's `local_resources` set) with **return-flow escape analysis**: at call sites that bind an allocator-style callee's result (`let x = some_op(...)` where `some_op` carries `Modify[result]`), substitute `result → x` in the inherited entry; then walk the operation's return expression to detect whether `x` flows out — when it does, lift its `Modify[x]` back to `Modify[result]` in the caller's row instead of discharging. The current filter handles the easy case (target is a local never referenced in the return); the new piece is the return-expression walk and the lift-back. Phase 1 of §Migration scopes the work.

**Multi-result operations.** For a named-tuple return like `make_pair() -> (a: Cell[Int], b: Cell[Int])`, the discharge analysis applies *per-projection*: each projection of `result` is its own bind site. The declared effects use `Modify[result.a]`, `Modify[result.b]` (relying on the existing named-tuple type from kernel spec §4.5 plus the field-access syntax of §6.7; 041 only widens `result` itself to be resolvable in effects rows). The inferred row can have partial coverage — e.g., when one projection escapes via the return and the other is consumed locally, only the first appears in the caller's row.

### Concrete examples

#### Local allocator use — discharged

```anthill
operation sum_via_accumulator(xs: List[Int]) -> Int
  let c = Cell.new(0)                            -- callee row: Modify[result]
                                                  -- bound to local c; c does not escape
  for_each(xs, lambda x -> Cell.set(c, Cell.get(c) + x))
                                                  -- Cell.set's Modify[c] is internal
  Cell.get(c)                                    -- pure
end
-- inferred row: ()  — empty; Modify[c] discharged because c never escapes the body.
```

#### Allocator result returned — propagates

```anthill
operation make_counter() -> Cell[Int] effects Modify[result]
  Cell.new(0)
end
-- inferred row: Modify[result] — c escapes via the return expression.
```

#### Allocator result passed onward — propagates as the caller's `Modify[result]`

```anthill
operation make_init_counter(initial: Int) -> Cell[Int] effects Modify[result]
  let c = Cell.new(initial)
  Cell.set(c, initial * 2)
  c
end
-- inferred row: Modify[result] — c escapes; Cell.set's internal Modify[c]
-- folds into the same target.
```

### Beyond in-memory allocation: external resources

The same `Modify[T]` machinery applies uniformly to any mutable resource with a first-class handle — whether the resource lives in process memory (`Cell`, `Map`, `Substitution`) or wraps an external connection (`Db`, `File`, `Socket`). A database connection is the canonical external case, and modeled **exactly like a mutable collection**: `open` allocates a fresh handle (Modify[result]), updates mutate through the handle (Modify[db]), reads are pure observation (like `Cell.get`).

```anthill
operation open(url: String) -> Db                                effects Modify[result]
operation close(db: Db) -> Unit                                  effects Modify[db]
operation update_user(db: Db, id: Int, name: String) -> Unit     effects Modify[db]
operation read_user(db: Db, id: Int) -> Option[User]             -- pure under current catalog
```

The discharge analysis runs identically to the Cell/Map case. If the operation owns the connection, all `Modify` effects discharge at the boundary:

```anthill
operation rename_user(url: String, id: Int, name: String) -> Unit
  let db = open(url)               -- Modify[result] discharged (db is local)
  update_user(db, id, name)        -- Modify[db] discharged
  close(db)                        -- Modify[db] discharged
  unit
end
-- inferred row: () — all Modify discharged; db never escapes.
```

If the caller passes in the connection, the update propagates:

```anthill
operation rename_user_with(db: Db, id: Int, name: String) -> Unit effects Modify[db]
  update_user(db, id, name)
end
-- inferred row: Modify[db].
```

This generalizes the design beyond in-memory allocators: `Modify[T]` is the effect of mutating any value-typed target — whether T is a fresh in-memory cell, a fresh database connection, or a connection the caller owns. The mechanism is one; only the resource changes.

**Why uniformity matters: pluggable backends.** The uniform `Modify[handle]` signature is what enables backend substitution. `anthill-todo`'s workitem store (see `examples/github-todo/docs/pluggable-backend.md`) is designed to swap between three implementations — local files, GitHub issues, and a future HTTP API — behind a single interface. All three must present identical effect signatures so they can be plugged in interchangeably. Splitting the effect into per-backend variants (e.g., separate `Modify[file]` for local vs `Modify[github_world]` for remote) would force every consumer of the interface to commit to one backend's effect shape, defeating the abstraction. The uniform `Modify[store]` captures the *mutating intent* regardless of where the state physically lives or who else can see it.

The type system tracks what's accessible through values that flow through it; whether other processes can also write to the same physical store is a runtime concern outside this model — the same abstraction `Cell.get` makes when declared pure even though its result depends on prior `Modify[c]` calls (pure relative to its own behavior, not in any stronger sense). Programs that need to reason about external concurrent writers add that as a separate effect or handler concern, outside this proposal's scope.

The database example is also a natural driver for the (currently absent) `Read[db]` precision discussed in §"Standard effect catalog": two reads of the same row are only redundant if no `Modify[db]` happens between them, and that dependency analysis is exactly what cell-precise `Read` would license. The current proposal stays at "reads declared pure" (matching `Cell.get`); a future driver — query planning, transactional-isolation reasoning, parallel-read safety — would lift this to `Read[db]` using the same scope-bound machinery.

### Discharge through higher-order combinators

The discharge rule above is stated for direct calls. Higher-order combinators (operations taking a function-typed parameter) inherit the callee's effect row by substitution. The substitution rule for `Modify[result]` follows the same "where does the value flow" analysis, lifted to function-typed parameters.

Let `f : (A) => B effect [Modify[result]]` be a function-typed parameter of operation `op`. When `op`'s body calls `f(x)` and binds the result to a local `r`, the inherited effect is `Modify[r]`. The discharge analysis then runs on `r` exactly as it would for a direct allocator call.

Three patterns cover the common cases:

| Combinator shape | result-relation | Inherited row |
|---|---|---|
| `apply(f, x)` (and `memoize(thunk)`) — caller's result IS f's result | `result_caller = result_f` | direct rebind: `Modify[result]` |
| `compose(g, f)` — f's result is consumed by g | depends on g's body | conservative union (see below) |
| `map(xs, f)` over a structural container | f's result enters an anonymous slot in the container | coarsen to container: `Modify[result]` |

**Direct rebind (`apply`, `memoize`):** the substitution `result_f → result_caller` is definitional — they refer to the same value. `memoize(lambda () -> Cell.new(0))` has row `Modify[result]` where `result` is `memoize`'s return slot.

**Container coarsening (`map`):** when f's result becomes an anonymous element of a structural return — built-in (`List`, `Set`, `Stream`) or user-defined (e.g., a custom `Tree[T]`) — the per-element targets are not separately addressable in the type. The inherited Modify coarsens to the container: `map(xs, Cell.new) : List[Cell[Int]] effects Modify[result]`. The trigger is purely structural and does *not* privilege stdlib sorts: a sort participates in coarsening whenever its declared shape exposes a recursive parametric position holding values of f's result type. Per-element precision (`∀ c ∈ result. Modify[c]`) would require row quantifiers — out of scope here.

**Conservative union (`compose`, `fold`):** when f's result is fed to another effectful callee whose signature doesn't expose whether the value escapes, the safe default is to union both rows and treat any Modify-on-fresh-target as potentially escaping. `compose(g, f)` with f allocating and g returning a fresh value has row `effect [E_f, E_g]`. A tighter analysis would need aliasing information that current signatures don't carry.

The rule is Modify-specific: see the standard effect catalog below for why other effect kinds propagate as-is without substitution.

### Standard effect catalog: scope-bound vs free

This proposal formalizes a small classification that the discharge rule depends on. The category boundary is **structural**: an effect is *scope-bound* when its target is a first-class value whose identity flows through the call graph — the discharge analysis ("does this target escape the body?") is then well-defined against that target. Effects without a value-identity target propagate without discharge.

| Category | Members | Discharge behavior |
|---|---|---|
| **Scope-bound** — value-typed target | `Modify[T]` (with T a parameter name or `result`) | Discharged at the function boundary when T doesn't escape. |
| **Source-bound** — sort-level or globally-implicit target | `Read[kb]` (sort-level, in use today), `NonDet`, `Random` (sketches) | Always propagates; no value-identity to discharge against. |
| **Handler-bound** — type parameter | `Error[E]` | Propagates until a handler catches it. |
| **Control** — no target | `Branch`, `Suspension` | Always propagates. |
| **Observable side-channel** — fixed external resource | `IO`, `Console` | Always propagates. |

Scope-bound effects participate in the discharge and HOF substitution analysis; the rest don't. The categorization is documented as part of 027's "standard effect catalog" charter (027 §Motivation point 2 already promises this).

**Modify is today's only scope-bound member; the category is extensible.** The discharge machinery built for `Modify[T]` — local flow tracking plus HOF substitution via direct rebind / container coarsening / conservative union — is not Modify-specific. It is the analysis "does this value's identity escape the body?" applied at a value-typed target. Any future effect with a value-typed target slots into the same analysis with no new mechanism. Candidates:

- **`Read[c]` with cell-precise target.** Not used today — `Read` currently targets sort-level resources like `kb`. The hook is open if a driver appears: parallel execution proving non-interference of disjoint reads; query dedup that needs to know two reads observe identical state; transactional-isolation reasoning over a database handle (see §"Beyond in-memory allocation"). Discharge is symmetric to `Modify[c]` — a `Read[c]` on a locally-allocated `c` that doesn't escape is discharged at the boundary, same machinery — and the HOF substitution rules transfer unchanged.
- **`Borrow[T]`** — Rust-style read/write borrow with sub-region semantics. Hooks in identically.
- **`Random[g]`** if RNG generators become first-class values rather than a global source.
- **`IO[h]`** if file handles or sockets get tracked precisely (sandboxing, capability passing).

Today's catalog has only `Modify[T]` in the scope-bound column because Read/Random/IO chose globally-implicit (or sort-level) targets, and Branch/Suspension/Error genuinely have no value-identity to discharge. That choice is per-effect and revisable per-effect — the discharge machinery doesn't depend on the current selection. The `(A) => B effect [E]` surface form stays unchanged because E is just a row of effect entries, each carrying its own kind-appropriate target precision.

### What this means for `Alloc[T]`

There is no `Alloc[T]` effect kind under this proposal. The earlier draft of 027.1 introduced one; that's now retired in favor of the unified `Modify[result]` framing. The unification:

- Removes one effect-kind from the standard catalog (less surface to learn).
- Reuses the existing `Modify[T]` handler-dispatch logic and the existing parameter-discharge analysis.
- Makes "allocation is fresh-target mutation" explicit at the type level.

### Variance

`Modify[T]` is covariant in T per 037's existing variance treatment. `Modify[result]` is variance-bound by the operation's return type — no special handling.

### Handler contract under 027

Allocator handlers (Cell-arena, Map-arena, Substitution-arena) are already registered for the `Modify` effect dispatch in `eval/effects.rs`. Under this proposal, they continue to dispatch on `Modify[T]` for the appropriate target sort. The distinction "this `Modify[T]` is on a fresh target vs an existing one" is invisible to the handler — it just allocates or mutates per the operation called. No handler-side changes.

## Revising 037

The following operations are revised:

| Operation | Old (037) | New |
|---|---|---|
| `Cell.new(initial: V) -> Cell[V]` | (no effects) | `effects Modify[result]` |
| `Map.empty() -> Map[K, V]` | (no effects) | `effects Modify[result]` |
| `Substitution.empty() -> Substitution` | (no effects) | `effects Modify[result]` |
| `Stream.new(...) -> Stream[T]` (when added) | (no effects) | `effects Modify[result]` |

037 §2 and §Identity each receive a one-line cross-reference pointing at 027.1 for the corrected effect declaration and the discharge rule. Larger 037 surgery is deferred to a 037.1 if/when more accumulates.

### Audit work

Small impact, since these ops are already called with parens in practice:

- Stdlib references to listed allocators: already parens-form, no call-site changes.
- Typer test fixtures: update to expect `Modify[result]` on allocator signatures.
- Internal helpers that today allocate transient Cells/Maps: discharge analysis removes the effect from their inferred row, so most of them gain an *empty* effect row — net win for readability.
- 037 cross-references: one line each in §2 and §Identity.

## Consequence for 039 (term-level constants)

Under 027.1, the const validator correctly rejects:

```anthill
const C: Cell[Int] = Cell.new(0)   -- rejected: Cell.new now has Modify[result]
```

The singleton-cell pattern needs an explicit surface. Recommendation: **stdlib `memoize[T]`** with memoization semantics:

```anthill
operation memoize[T, E](thunk: () -> T effects E) -> T effects E
  -- Evaluates `thunk` on first call, caches the result, returns the cache thereafter.
  -- The thunk's effects (e.g. Modify[result]) execute exactly once.
```

Singleton cell:

```anthill
operation counter() -> Cell[Int] effects Modify[result]
  memoize(() -> Cell.new(0))
end
```

Two alternatives — `lazy const` keyword, or a dedicated `static` declaration — are rejected: they grow language surface for a niche pattern that composes adequately from `memoize` + ordinary operations.

The `memoize` operation itself is a small stdlib WI, separate from 027.1. Effect-row polymorphism in E is already supported by the type system (see `stdlib/anthill/prelude/function.anthill`'s `apply` and the Stream sort in `stream.anthill`, both with `effects E` parameters); the rest is a one-cell memoization slot.

## Alloc failure — out of scope here, opt-in via parallel ops

Some contexts (embedded, OS, HFT) need to handle allocation failure explicitly. The default `Cell.new` / `Map.empty` shapes above are infallible (matches reality for typical hosted code; matches `Box::new` in Rust, `new` in Java, `ref` in OCaml).

Code that cares uses parallel `try_*` operations, separately declared:

```anthill
operation try_new(initial: V) -> Result[Cell[V], OutOfMemory]
  effects Modify[result], Error[OutOfMemory]
```

Following the precedent of `Box::new` / `Box::try_new` in Rust and `new` / `new(std::nothrow)` in C++. Per-call-site opt-in keeps the default surface clean while letting fallibility-aware code make the choice explicit. Out of scope for 027.1 — a follow-up stdlib WI adds the `try_*` variants once a concrete driver appears.

## Migration

**Phase 0 — Land 041 (operation result naming).** `result` becomes reserved across the whole operation declaration; named-tuple result syntax becomes available for multi-output ops. Prerequisite.

**Phase 1 — Add return-flow escape analysis.** Two pieces of new typer work: (a) at call sites that bind an allocator-style callee's return (`let x = some_op()` where `some_op` has `Modify[result]` in its row), substitute `result → x` in the inherited entry; (b) walk the operation's return expression to detect whether `x` flows out — lift `Modify[x]` back to `Modify[result]` when it does, otherwise discharge. The existing `is_local_resource` filter (symbol-name membership) becomes the "discharge" branch of step (b); the "lift" branch is genuinely new. Tests: the four escape cases (returned, stored-then-escaping, bound-and-local, captured-by-closure), the discharged/propagated examples above, a multi-result partial-discharge case (one projection escapes via the return, the other is consumed locally), and a user-defined-container coarsening case (e.g., a custom `Tree[T]` paralleling List/Stream — confirms structural trigger, not stdlib-privileged). Error message shape when escape is detected but the signature doesn't list the corresponding Modify: *"this operation's body allocates a `Cell[V]` that escapes via [path]; either add `effects Modify[result]` to the signature, or change the body to keep the allocation local."*

**Phase 2 — Revise 037 allocators.** Update `Cell.new`, `Map.empty`, `Substitution.empty` signatures to `effects Modify[result]`. Audit stdlib + examples. Update typer test fixtures. Cross-reference from 037 §2 and §Identity.

**Phase 3 — `memoize` operation.** Add `memoize[T, E]` to stdlib with memoization semantics. Convert any code that depended on the implicit singleton semantics from 037-as-it-stands (likely none).

**Phase 4 — 039 validator integration.** Once phases 1–2 land, 039's validator naturally rejects `const C: Cell[Int] = Cell.new(0)` — no extra work needed in 039.

Each phase lands independently with tests. Phase 0 (WI-261, proposal 041) is the only hard prerequisite for the rest.

## Open questions

1. **HOF inheritance: sharpness vs simplicity.** §"Discharge through higher-order combinators" gives three patterns — direct rebind, container coarsening, conservative union. The conservative-union case (`compose`, `fold`) over-reports when the intermediate value provably doesn't escape. Tightening it requires aliasing information at the signature level (something like "this function's result does/doesn't alias its argument"), which is a significantly larger addition. **Suggested resolution**: ship the conservative rule; revisit only if a concrete driver appears where the over-report blocks useful code. Region-polymorphic effects (à la Tofte–Talpin) are the standard precedent if/when that day comes.

   **Closure-capture discharge is the same question, redirected.** When a closure that captures a `Modify` target is passed to a callee whose signature doesn't expose whether the closure escapes (stored long-term vs invoked synchronously and discarded), the analysis must conservatively propagate any captures — even when the callee in fact invokes the closure synchronously. The basic rule is mechanical: closure escape → captured-target escape (a local captured by an escaping closure lifts to `Modify[result]` in the enclosing op); the only genuine fork is whether to inspect callee disposition, which is this same conservative-vs-sharp tradeoff. Sharper analysis would need callee-disposition annotations — the anthill analog of Rust's `Fn` / `FnMut` / `FnOnce`, or an explicit "consumed synchronously" marker on the parameter. Same resolution: conservative until a driver justifies the addition.


## Out of scope

- Concrete source-bound effects beyond today's catalog: `Read[T]` cell-precision, `NonDet`, `Random`, and the operations that would use them (`rand`, `currentTime`, IO sources). Sketched in §"Standard effect catalog"; needs a concrete driver (likely `rand` in stdlib) before any of it lands.
- Effect handler implementations — falls under 037's arena machinery (existing) and 027's handler framework (existing).
- `lazy const` / `static` surface forms — discussed and rejected.
- 037's arena vs per-sort-arena design — falls under 037.
- Call-site rule for 0-arg ops — already in [039 §Call-site sugar](039-term-level-constants.md).
- Operation result naming surface — covered by [041](041-operation-result-naming.md).
- `try_*` fallible allocator variants — separate stdlib WI when an embedded/HFT driver appears.
- Rust codegen mapping for `Modify[result]` and discharge — covered in `docs/rust-forward-mapping.md` §5.6–5.7. The connection is non-trivial: this proposal's Phase 1 discharge analysis directly determines whether allocated values become local Rust variables or owned returns. Tight discharge → tighter generated Rust signatures (fewer heap allocations, more stack-resident values). Anthill's discharge is structurally the same computation Rust's borrow checker performs for lifetimes, applied to effect targets rather than references.
