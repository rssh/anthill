# Proposal 027.1: Allocator Effects via `Modify[result]` and Local Discharge

**Status:** Draft
**Depends on:** [027-effect-handlers-and-standard-effects](027-effect-handlers-and-standard-effects.md) (handler model + standard effect catalog), [037-anthill-state-model](037-anthill-state-model.md) (Cell, allocator operations), [041-operation-result-naming](041-operation-result-naming.md) (one-arm typer extension making `result` resolvable in effects rows; WI-261)
**Related:** [039-term-level-constants](039-term-level-constants.md) (const validator inherits the dishonesty if allocators stay pure), [026-expression-evaluator](026-expression-evaluator.md) (CSE / inlining / folding need an honest effect declaration)
**Affects:** `stdlib/anthill/prelude/cell.anthill` + `map.anthill` + `substitution.anthill` (allocator signatures), 037 §2 and §Identity (one-line cross-reference), typer (extend the existing `Modify[c]` discharge logic to cover `Modify[result]`)

## Motivation

Proposal 037 §2 declares allocator operations as effect-pure:

> Construction is allocation, not mutation — it doesn't modify any existing Cell, so it doesn't carry `Modify[anything]`. […] `Map.empty()`, `Substitution.empty()`, `List.nil()` are all effect-pure even though each call yields a fresh handle.

And 037 §Identity says:

> Two calls to `Cell.new(0)` produce two distinct cells even with identical initial values. Identity is allocation-time, not value-time.

Jointly: an operation declared with no effects whose runtime semantics break equational reasoning. This stance — *pure-but-identity-distinguishing* — is rare in the algebraic-effects literature. Most systems either declare the operation effectful (Haskell IO/ST, Koka `heap`, Frank / Eff handler-defined) or accept the impurity without an effect system (OCaml). 037 took a fourth path: pragmatically defensible (keeps allocator call sites lightweight), but it leaves the effect system materially less useful than it should be.

Two pressures push the revision now:

1. **039's const validator needs honest input.** A `const` body must be "foldable and effect-free." If `Cell.new` is declared with no effects, the validator admits `const C: Cell[Int] = Cell.new(0)` — accidentally producing a singleton-cell semantics that depends on a fragile invariant the user has to know. The right answer is for `Cell.new`'s declaration to forbid this admission outright.

2. **026's evaluator transformations will mis-rewrite.** As load-time folding, CSE, and inlining mature (and especially as 026 M5+ adds Suspension / Branch state handling), the effect annotation becomes load-bearing. Today's "pure-but-distinguishing" allocator lets a CSE pass collapse `(Cell.new(0), Cell.new(0))` to `let x = Cell.new(0); (x, x)` — silently wrong.

And a third pressure becomes obvious once you start counting allocators in real code:

3. **Pollution.** Almost every nontrivial operation allocates a transient Cell, Map, or Substitution. A naive "every allocator carries its own effect kind" stance would put `Alloc[...]` lines in the effect row of nearly every signature in the codebase. Caller signatures fill with allocations they don't care about and can't observe. The effect system stops being read.

This proposal addresses all three by unifying allocation into the existing `Modify[T]` machinery and adding a **local-body-flow discharge rule** that already exists for `Modify[c]` on input parameters — extended to cover `Modify[result]` on returns.

## Design

### Allocator effects collapse into `Modify[result]`

There is no new `Alloc[T]` effect kind. Allocator operations declare:

```anthill
operation new(initial: V) -> Cell[V] effects Modify[result]
operation set(c: Cell[V], v: V) -> Unit effects Modify[c]
operation get(c: Cell[V]) -> V                                -- pure
```

`result` is the reserved name for the operation's return slot, introduced by [proposal 041](041-operation-result-naming.md) and already reserved for `ensures` clauses in the kernel spec (§5.4).

**Why this unifies cleanly.** Allocation isn't a fundamentally distinct kind of effect — it's the introduction of state with a fresh identity. Setting `c` to `v` mutates an existing target whose identity is shared with the caller; constructing a fresh `Cell` mutates a target whose identity is, momentarily, not shared with anyone. The mechanism is the same; only the target source differs:

| Effect occurrence | Target source | Identity behavior |
|---|---|---|
| `Modify[c]` on input parameter `c` | Caller-provided | Caller observes the modification |
| `Modify[result]` on return | Fresh, owned by callee until returned | Caller observes only if they bind the result |

The 037 lines that argue "allocation isn't mutation" remain correct in their narrow sense — allocator ops don't mutate any *pre-existing* cell. But they do introduce mutable state, and that's enough to break referential transparency, which is what an effect declaration captures.

### Discharge rule — local body flow, same as `Modify[c]`

The typer already does the analysis needed. For `Modify[c]` where `c` is an input parameter, the propagation rule is: if the body invokes `Modify` on a value that traces back to `c` along the body's data flow, the operation's signature must declare `Modify[c]`; the caller substitutes the actual argument for `c` at the call site.

**Extension for `Modify[result]`:** treat `result` as an additional bind site whose flow originates at the return expression. The discharge rule is the dual:

> An operation that calls a callee with `Modify[result]` in its row inherits `Modify[x]` for the local name `x` it binds the result to. If `x` does not escape the operation's body, the effect is discharged at the function boundary.

"Escape" is determined by the same local flow analysis already used for Modify-on-parameter:

- Returned in the operation's return expression → escapes → propagates.
- Stored in a fresh value that itself escapes → escapes → propagates.
- Bound by a `let` and only consumed by other operations whose effects don't re-export it → does not escape → discharged.
- Passed to a closure that captures and returns it → escapes → propagates.

No new analysis; same machinery, one more bind site.

### Concrete examples

#### Local allocator use — discharged

```anthill
operation sum_via_accumulator(xs: List[Int]) -> Int
  let c = Cell.new(0)                            -- callee row: Modify[result]
                                                  -- bound to local c; c does not escape
  for_each(xs, lambda x -> Cell.set(c, Cell.get(c) + x))
                                                  -- Cell.set's Modify[c] is internal
  Cell.get(c)                                    -- pure
end
-- inferred row: ()  — empty; Modify[c] discharged because c never escapes the body.
```

#### Allocator result returned — propagates

```anthill
operation make_counter() -> Cell[Int] effects Modify[result]
  Cell.new(0)
end
-- inferred row: Modify[result] — c escapes via the return expression.
```

#### Allocator result passed onward — propagates as the caller's `Modify[result]`

```anthill
operation make_init_counter(initial: Int) -> Cell[Int] effects Modify[result]
  let c = Cell.new(initial)
  Cell.set(c, initial * 2)
  c
end
-- inferred row: Modify[result] — c escapes; Cell.set's internal Modify[c]
-- folds into the same target.
```

### Standard effect catalog: scope-bound vs free

This proposal also formalizes a small classification that the discharge rule depends on:

| Effect kind | Discharge behavior |
|---|---|
| `Modify[T]` (with T a parameter name or `result`) | **Scope-bound.** Discharged at the function boundary when T doesn't escape. |
| `Read[T]`, `NonDet`, `Random` (out of scope here — sketch only) | **Source-bound.** The source is implicitly global; always propagates. |
| `Error[E]` | **Handler-bound.** Propagates until a handler catches it. |
| `Branch`, `Suspension` | **Control.** Always propagates. |
| `IO`, `Console` | **Observable side-channel.** Always propagates. |

Scope-bound effects participate in the discharge analysis; the rest don't. The categorization is documented as part of 027's "standard effect catalog" charter (027 §Motivation point 2 already promises this).

### What this means for `Alloc[T]`

There is no `Alloc[T]` effect kind under this proposal. The earlier draft of 027.1 introduced one; that's now retired in favor of the unified `Modify[result]` framing. The unification:

- Removes one effect-kind from the standard catalog (less surface to learn).
- Reuses the existing `Modify[T]` handler-dispatch logic and the existing parameter-discharge analysis.
- Makes "allocation is fresh-target mutation" explicit at the type level.

### Variance

`Modify[T]` is covariant in T per 037's existing variance treatment. `Modify[result]` is variance-bound by the operation's return type — no special handling.

### Handler contract under 027

Allocator handlers (Cell-arena, Map-arena, Substitution-arena) are already registered for the `Modify` effect dispatch in `eval/effects.rs`. Under this proposal, they continue to dispatch on `Modify[T]` for the appropriate target sort. The distinction "this `Modify[T]` is on a fresh target vs an existing one" is invisible to the handler — it just allocates or mutates per the operation called. No handler-side changes.

## Revising 037

The following operations are revised:

| Operation | Old (037) | New |
|---|---|---|
| `Cell.new(initial: V) -> Cell[V]` | (no effects) | `effects Modify[result]` |
| `Map.empty() -> Map[K, V]` | (no effects) | `effects Modify[result]` |
| `Substitution.empty() -> Substitution` | (no effects) | `effects Modify[result]` |
| `Stream.new(...) -> Stream[T]` (when added) | (no effects) | `effects Modify[result]` |

037 §2 and §Identity each receive a one-line cross-reference pointing at 027.1 for the corrected effect declaration and the discharge rule. Larger 037 surgery is deferred to a 037.1 if/when more accumulates.

### Audit work

Small impact, since these ops are already called with parens in practice:

- Stdlib references to listed allocators: already parens-form, no call-site changes.
- Typer test fixtures: update to expect `Modify[result]` on allocator signatures.
- Internal helpers that today allocate transient Cells/Maps: discharge analysis removes the effect from their inferred row, so most of them gain an *empty* effect row — net win for readability.
- 037 cross-references: one line each in §2 and §Identity.

## Consequence for 039 (term-level constants)

Under 027.1, the const validator correctly rejects:

```anthill
const C: Cell[Int] = Cell.new(0)   -- rejected: Cell.new now has Modify[result]
```

The singleton-cell pattern needs an explicit surface. Recommendation: **stdlib `once[T]`** with memoization semantics:

```anthill
operation once[T, E](thunk: () -> T effects E) -> T effects E
  -- Evaluates `thunk` on first call, caches the result, returns the cache thereafter.
  -- The thunk's effects (e.g. Modify[result]) execute exactly once.
```

Singleton cell:

```anthill
operation counter() -> Cell[Int] effects Modify[result]
  once(() -> Cell.new(0))
end
```

Two alternatives — `lazy const` keyword, or a dedicated `static` declaration — are rejected: they grow language surface for a niche pattern that composes adequately from `once` + ordinary operations.

The `once` operation itself is a small stdlib WI, separate from 027.1. Its tricky bit is effect-row polymorphism in E; the rest is a one-cell memoization slot. (If effect polymorphism isn't expressible yet, a temporary monomorphic version covers the immediate singleton-cell case.)

## Alloc failure — out of scope here, opt-in via parallel ops

Some contexts (embedded, OS, HFT) need to handle allocation failure explicitly. The default `Cell.new` / `Map.empty` shapes above are infallible (matches reality for typical hosted code; matches `Box::new` in Rust, `new` in Java, `ref` in OCaml).

Code that cares uses parallel `try_*` operations, separately declared:

```anthill
operation try_new(initial: V) -> Result[Cell[V], OutOfMemory]
  effects Modify[result], Error[OutOfMemory]
```

Following the precedent of `Box::new` / `Box::try_new` in Rust and `new` / `new(std::nothrow)` in C++. Per-call-site opt-in keeps the default surface clean while letting fallibility-aware code make the choice explicit. Out of scope for 027.1 — a follow-up stdlib WI adds the `try_*` variants once a concrete driver appears.

## Migration

**Phase 0 — Land 041 (operation result naming).** `result` becomes reserved across the whole operation declaration; named-tuple result syntax becomes available for multi-output ops. Prerequisite.

**Phase 1 — Extend the discharge analysis.** Teach the typer to treat `result` as a bind site for `Modify[result]` discharge, parallel to the existing parameter-name discharge. Add tests for both the discharged and propagated cases (the two non-trivial examples above).

**Phase 2 — Revise 037 allocators.** Update `Cell.new`, `Map.empty`, `Substitution.empty` signatures to `effects Modify[result]`. Audit stdlib + examples. Update typer test fixtures. Cross-reference from 037 §2 and §Identity.

**Phase 3 — `once` operation.** Add `once[T, E]` to stdlib with memoization semantics. Convert any code that depended on the implicit singleton semantics from 037-as-it-stands (likely none).

**Phase 4 — 039 validator integration.** Once phases 1–2 land, 039's validator naturally rejects `const C: Cell[Int] = Cell.new(0)` — no extra work needed in 039.

Each phase lands independently with tests. Phase 0 (WI-261, proposal 041) is the only hard prerequisite for the rest.

## Open questions

1. **Discharge precision on closures.** If a Cell handle is captured by a closure and the closure escapes, the discharge analysis must conservatively propagate `Modify[result]`. Edge cases worth pinning down with tests: closures stored in long-lived containers; closures returned via tuples; closures passed back through a callback. **Suggested resolution**: lean conservative — when in doubt, propagate. The user can refactor to force discharge.

2. **Discharge for multi-result operations.** A named-tuple-returning op like `make_pair() -> (a: Cell[Int], b: Cell[Int])` declares `effects Modify[result.a], Modify[result.b]` — using the existing named-tuple type (§4.5) and the existing field-access syntax (§6.7); 041 only widens `result` to be resolvable in effects rows. The discharge analysis applies per-projection. Worth a test demonstrating partial discharge — one result escapes, the other doesn't.

3. **Effect-row inference for `once`.** Effect-polymorphic `once[T, E]` — confirm expressible in the current type system. If not, monomorphic stopgap.

4. **`Read[T]` / `NonDet` follow-up.** Still out of scope for 027.1; sketch in §"Standard effect catalog" remains the design preview. Needs a concrete driver (likely `rand` in stdlib).

5. **Discharge failure surface.** When the typer cannot discharge (escape detected, but caller's signature doesn't list the corresponding Modify), what's the error? Probably "this operation's body allocates a `Cell[V]` that escapes via [path]; either add `effects Modify[result]` to the signature, or change the body to keep the allocation local." Friendly enough.
