# Proposal 043.1 — Compile-time macros: `[simp]` rules that rewrite via syntax→syntax operations

**Status:** Draft (2026-07-15)
**Extends:** [043-simp-rewrite](043-simp-rewrite.md) — its `[simp]` engine. Macros are not a bolt-on: they fall out of that engine organically (§2). 043 §4.2 already classifies builtins by *argument* domain (expression- vs value-accepting); this reads the *result* domain the same way, and opens the door to user code.
**Depends on:** 043's `[simp]` engine (typer-side occurrence rewriter — WI-276/277 occurrence matching + build side, delivered; the fire site `kb/simp_rewrite.rs::try_fire`), the reflect `NodeOccurrence` / `Expr` substrate (`reflect.anthill` §Occurrences + the `apply`/`lambda_expr`/`dot_apply`/… entities), the compile-time-eval-of-a-bodied-op precedent (WI-625 scratch interpreter), and the **typed-occurrence** substrate — *delivered* as each occurrence's typer-stamped `inferred_type` (WI-502; macros read arg types via it, §3.4).
**Driving client:** [052-rules-as-stream-valued-operations](052-rules-as-stream-valued-operations.md) / WI-714 — the `where`/`join` **row-lambda crossing** (§4).

---

## 1. Motivation

043 §4.2 sorts builtins by **argument domain**: a builtin that accepts **expressions** — `min_sort`, `find_operation_on_sort`, `constant_fold`, the reflect/`Expr` builtins — runs at **compile time**, over the matched occurrence; a value-domain builtin runs at evaluation. But 043 keeps this bridge **engine-provided** and never enumerates which builtins constitute it or how user code reaches it. There is no way for a user to author a compile-time **syntax → syntax** transform.

WI-714's `where`/`join` need exactly that. A filter is a **row lambda**, `where(r, c -> eq(c.x, 1))`, and must become the query `guarded(r.query, eq(?col_x, 1))` — a transform that (a) reads the lambda **body as syntax**, (b) maps `c.x` to the relation's column logic-var, (c) maps `eq(…)` to a **goal atom**. This is neither a template rewrite (it *computes*: a schema lookup, a goal construction) nor a runtime op (the lambda must **never be applied** — at query-build time the column is an unbound logic var, so `eq(?col_x, 1)` has no `Bool`). It is a **macro**: the row lambda is a syntax bracket, and the transform consumes its body.

**Baseline** (verified this session): a user op returning an occurrence, placed in a `[simp]` RHS, is *not* run — `substitute_to_occurrence` (`simp_rewrite.rs`) builds the RHS purely by template substitution, and the op is left as an ordinary expression (its args typed as values). So the capability is genuinely absent.

## 2. What this adds — a macro, read from the signature

A **macro** is a **syntax→syntax** operation: every parameter is an occurrence (`NodeOccurrence` / reflect `Expr`) *and* so is the result. When one appears in a fired `[simp]` rule's RHS, the engine **evaluates it at compile time** and splices the returned occurrence as the rewritten node — consuming the matched syntax, producing new syntax.

There is **no new concept and no marker** — the *signature* classifies it, and it lines up exactly with 043 §4.2's two-sided domain view. The **argument** side is 043 §4.2 verbatim (a builtin that *accepts expressions* runs at compile time, while the expression — not a value — is in hand); the **result** side (it *produces* an occurrence) is what makes it a **rewrite** that gets spliced, rather than a guard-reader that accepts an expression but returns a value. So `guarded_of(occ, occ) -> occ` is a macro; `dbl(x) -> Int64` (value in *and* out) is an ordinary runtime op; `min_sort(occ) -> Sort` (expr in, value out) is a compile-time *guard* reader, not a macro. The *same* op is an ordinary function at a runtime call site and a macro only **as the RHS of a `[simp]` rule**. 043's purity/termination discipline (§4.5) applies unchanged: **macros are pure and terminating.**

## 3. The mechanism

**3.1 Classification.** An op is a macro iff it is **syntax→syntax**: every parameter is an occurrence (`NodeOccurrence` / reflect `Expr`) *and* the result is one too — no attribute, no keyword. Both conditions carry weight:

- **Args are occurrences** — a macro is fed the matched **pattern-var occurrences**, so a value parameter would type-mismatch when handed one (exactly this session's probe). This *is* 043 §4.2's argument-domain rule.
- **Result is *exactly* an occurrence** — not a container of one (`Option[NodeOccurrence]`): the result is **spliced**, so nothing else could serve. This is what separates a macro from a compile-time *guard* reader (`min_sort(occ) -> Sort`: expr in, value out — not spliced).

An op that should *not* macro-expand has a non-occurrence in **either** position — an ordinary op — which a macro reaches by **emitting a call to it** (rewrite-to-it), never by wrapping. A load-time purity check keys on the same set (§6).

**3.2 Firing** (in `simp_rewrite.rs::try_fire`, after the LHS matches). Today:
```
if let Some(subst) = kb.match_view(lhs, occ) {                          // LHS matched, σ bound
    return Some(substitute_to_occurrence(kb, rhs, &subst, occ, pass));  // template only
}
```
`substitute_to_occurrence` already reuses each matched pattern var's child occurrence in place, so an RHS `f(?a, ?b)` builds `apply(f, [<occ a>, <occ b>])` — the args are the **real occurrences**. The new step: when `f` is a macro (result is an occurrence),
- **evaluate** `f` at compile time via a scratch interpreter (WI-625 precedent), binding its params to the argument occurrences as `Value::Node`, running its body, obtaining a `Value::Node` result;
- **splice** that occurrence as the rewrite result and **re-type it** — a created occurrence must be typed before anything else reads it, so the engine runs `type_check_node` on the spliced subtree (043 design Appendix A already makes the typer re-entrant for exactly this), giving its new nodes their own `type_of` so downstream (further simp, `req_insertion`, eval, IR) sees a typed tree;
- then continue the bottom-up rewrite (the output may hold further redexes), **fuel-bounded** (043 §4.5) with the `Synthesized.from` ancestor-loop check.

**3.3 What a macro body may do.** **Read** its argument occurrences — structure via `occurrence_expr(occ) -> Expr` (dispatch on the `apply`/`lambda_expr`/`dot_apply`/`if_expr`/… variants, children as `NodeOccurrence`), type via the stamped `inferred_type`, the KB via ambient `kb()` (§3.4). **Build** its result through the occurrence-building API below. Pure; it may raise a **compile-time `Error`** (a diagnostic at the redex span, via the `Synthesized` chain), but performs no IO or state mutation.

The **build side is new**: the Rust primitives exist (`NodeOccurrence::new_expr` / `synthesized_expr`) but are unexposed, so 043.1 adds the symmetric anthill surface — a reader `occurrence_expr(occ) -> Expr` and a builder `make_occurrence(e: Expr, from: NodeOccurrence) -> NodeOccurrence` (its `Expr` children are `NodeOccurrence`s; `from` is the source occurrence the built node points at for diagnostics — §3.5 — exactly the `from` `synthesized_expr` already carries, so a macro sets provenance by picking an input occurrence, never a raw span). `make_fn`/`as_term` build `Term`s, *not* occurrences — a macro returns a `NodeOccurrence`.

**3.4 Context — a macro runs over *typed* occurrences.** Raw syntax alone is not enough: `guarded_of` must read `r`'s **schema** (its columns) to map `c.x`, and that lives in `r`'s *type*, not its syntax. A macro therefore runs after its arguments are typed — the typer-side rewriter is bottom-up (043 design §5.2), so each argument occurrence is already typed when the macro fires — and reads each argument's type from the occurrence's stamped **`inferred_type`** — the type the typer already writes on every `Expr` occurrence (`node_occurrence.rs`; `typing.rs` `set_inferred_type`; **WI-502** typed-term substrate; carried through rebuilds), surfaced to the macro via a thin reflect reader (a `type_of`-style op, Proposal 022). *(043's design note that the type is discarded — §5.4 — is stale; `inferred_type` retains it. Probed this session.)* Anything further (resolve `cells` to its rule, read a fact) is a KB query via the ambient `kb()` reflect op, which already returns the current KB in *both* phases (compile-time and runtime). So a macro's context is **its arguments' types + the ambient KB**, supplied *ambiently* — not reconstructed, and not threaded as an argument (§7).

**3.5 Diagnostics — provenance is threaded; *precision* is the macro's to give.** A macro never manages raw spans, but it *does* choose what each built node points at. Every synthesized node carries `Synthesized{from, by}`, where **`from`** is a source occurrence the build API takes (§3.3). Reused input occurrences keep their own spans; for a *new* node the macro passes the relevant **parsed sub-part of an argument** as `from` — e.g. the original `c.x` occurrence for the guard var it becomes — so a failure (during expansion, or in the re-typed generated code) localizes to that precise source, not merely the whole `where(…)` redex. The engine threads the chain and renders the trace; the macro's one job is to attribute each built node to the input part it came from — attribute nothing and the default `from` is the redex (coarse but correct).

## 4. The `where`/`join` client (WI-714)

```anthill
rule where(?r, ?cond) <=> guarded_of(?r, ?cond)   [simp]

operation guarded_of(r: NodeOccurrence, cond: NodeOccurrence) -> NodeOccurrence
  -- a macro (it returns an occurrence): read cond as `lambda_expr(param, body)`,
  -- read r's schema (its columns), rewrite each `param.field` in body to that
  -- column's logic var and each `eq/lt/…(…)` to a goal atom, and return
  -- `guarded(query_of(r), <goal>)`.
```

`join` is the same shape with **two** row binders → `conjunction`, merging columns. The lambda is never applied — it is the syntax bracket that (a) scopes the columns so `c.x` type-checks against the schema (052 §"Relational algebra": the binder is what makes the column resolvable) and (b) delivers its body to the macro. This is the "lambda crossing" 052 needs, and why this proposal is a **prerequisite** for those increments.

## 5. Relationship to 043's engine

Same **one rewriter** (043 §4.7), same two call sites — but macro evaluation is **occurrence-side only** (the typer call site), because a macro consumes occurrences. The logical-engine (term) call site does not run macros: a runtime `Term` goal is not a syntax bracket (043 §1's expression-vs-value boundary). The value→expression wall stands; a macro is a compile-time syntax transform, never a runtime call.

## 6. Termination & purity

A macro runs during compilation, so it must be **pure and terminating**. Non-termination is bounded by simp fuel (loud on exhaustion) plus the ancestor-loop check. A macro that raises `Error` becomes a compile-time diagnostic at the redex span. A **load-time check** keys on the classifier (§3.1): an occurrence-returning op must have a pure (at most `Error`) effect row.

## 7. Open questions

- **Occurrence-building API shape** — a single `make_occurrence(e: Expr)` symmetric to `occurrence_expr` (lean) vs per-shape builders (`make_apply`, `make_dot_apply`, …). The *decision* to expose an anthill build API (over Rust-only host-builtin macros) is settled — §3.3, §8.
- **Typed-occurrence substrate — resolved (probed).** Occurrences already carry the typer-stamped `inferred_type` (WI-502; §3.4), so WI-720 only adds a thin reflect reader over it — no substrate prerequisite, no hidden blocker.
- **Context surface.** *Semantic* context (arg types, KB) is ambient — `type_of` + `kb()` (phase-adaptive). *Diagnostic* provenance is the build API's `from` (§3.5), not a separate surface. So neither needs a bespoke context API. The one genuinely-open bit: does a macro ever need the *enclosing typing env* (Γ / the redex-visible binders) beyond arg types — a `current_context()` generalization? `where`/`join` do not.

*(Non-questions, resolved above, not listed: **result-domain** — the result must be exactly `NodeOccurrence`, since it is spliced (§3.1); **argument delivery** — args are the matched occurrences, handed over whole (§3.2); **`[unfold]` interaction** — a macro's output just keeps rewriting under 043 §4.5/§4.6, no macro-specific rule.)*

## 8. Non-goals

- **The build API is not WI-189 quote — it is the *primitive* WI-189 would lower to.** `make_occurrence` / `occurrence_expr` are the explicit AST layer: name every symbol, no hygiene, no lexical capture, no splice. WI-189 `↑{…}`/`${…}` is *surface sugar + staging/hygiene* that desugars *into* this API (`↑{ eq(c.x, 1) }` → `make_occurrence(apply(eq, [dot(…), int_lit(1)]))`). So the build API is a **substrate** WI-189 could build on, not a competitor; a macro manipulates the concrete symbols already in its input occurrences (not fresh hygienic binders), so it needs only the primitive — WI-189 stays optional future ergonomics, never a prerequisite.
- No grammar change.
- No side-effecting macros — pure, per 043.
- Not the term-side (runtime) rewriter — macros are compile-time, occurrence-side.

## 9. Implementation sketch (for the prerequisite WI)

1. **Classifier** — `KnowledgeBase::is_macro(op)` = declared result is an occurrence (§3.1, §7).
2. **Fire hook** — in `simp_rewrite.rs::try_fire`, after `substitute_to_occurrence`, detect a macro-head result and route it through compile-time eval.
3. **Compile-time eval + re-type** — reuse the WI-625 scratch interpreter; bind occurrence args as `Value::Node`; run the body; splice the `Value::Node` result; **re-type the spliced subtree** (`type_check_node`, 043 Appendix A); continue fuel-bounded.
4. **Purity check** — load-time, on the classified set (§6).
5. **Occurrence read/build API** — expose `occurrence_expr(occ) -> Expr` (reader), `make_occurrence(e: Expr, from: NodeOccurrence) -> NodeOccurrence` (builder over `synthesized_expr`; `from` = diagnostic source, §3.5), and a reader over the stamped `inferred_type` (`occurrence_type(occ) -> Option[Type]`); wire ambient `kb()`. (`current_context` is a later generalization, §7.)
6. **Tests** — a minimal occurrence-returning op that rewrites a marker occurrence at compile time (the inverse of this session's negative baseline); then the `where` macro end-to-end.
